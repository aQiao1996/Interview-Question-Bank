# 系统设计

这里用于整理系统设计相关面试题，后续按限流、熔断、降级、缓存一致性、分布式锁、消息队列、幂等、链路追踪、高并发读写等方向追加。

## 1、系统为什么需要限流，常见限流算法有哪些
限流用于限制单位时间内的请求数量，防止流量突增打垮服务。它常用于接口保护、登录防刷、短信发送、支付下单、网关入口等场景。

::: details 详情
### 为什么需要限流

如果没有限流，突发流量可能导致：

- CPU 或内存打满。
- 数据库连接池耗尽。
- 下游服务被压垮。
- 队列积压。
- 正常用户请求被拖慢。

限流的目标不是让所有请求都成功，而是在系统承载范围内保护核心链路。

### 固定窗口算法

把时间按固定窗口切分，例如每分钟最多 100 次。

优点：实现简单。

缺点：窗口边界可能出现流量突刺。

### 滑动窗口算法

记录更细粒度的时间片或请求时间，按最近一段时间统计请求数。

优点：比固定窗口更平滑。

缺点：实现和存储成本更高。

### 漏桶算法

请求进入桶中，按照固定速率流出。

优点：输出流量稳定。

缺点：突发流量处理能力较弱。

### 令牌桶算法

系统按固定速率生成令牌，请求必须拿到令牌才能通过。

优点：既能限制平均速率，也允许一定突发流量。

### 面试要点

- 限流可以放在网关、服务端、用户维度、IP 维度或业务维度。
- 高并发分布式场景常结合 Redis 或网关限流。
- 限流后要返回明确错误码，并配合降级、排队或重试提示。
- 限流阈值需要结合容量压测和业务优先级确定。
:::

## 2、熔断和降级有什么区别
熔断和降级都是系统稳定性手段。熔断更关注在下游异常时快速切断调用，避免故障扩散；降级更关注在资源不足或功能异常时牺牲非核心能力，保证核心链路可用。

::: details 详情
### 熔断

熔断类似电路保险丝。

当某个下游服务错误率或超时率过高时，上游不再继续请求它，而是直接返回失败或兜底结果。

常见状态：

- closed：正常调用。
- open：熔断打开，直接拒绝或兜底。
- half-open：放少量请求探测下游是否恢复。

### 降级

降级是主动降低服务能力，例如：

- 首页不展示推荐模块。
- 搜索只返回缓存结果。
- 评论区暂时关闭。
- 非核心接口返回默认值。
- 暂停低优先级任务。

降级可以是自动触发，也可以由运营或研发手动开关触发。

### 二者区别

- 熔断通常针对下游依赖异常。
- 降级通常针对整体资源压力或非核心能力取舍。
- 熔断强调保护调用链。
- 降级强调保核心、舍非核心。

### 组合使用

实际系统中二者经常一起使用：

```txt
下游推荐服务超时
-> 熔断推荐服务调用
-> 降级为默认推荐列表
```

### 面试要点

- 熔断需要指标：错误率、超时率、并发数等。
- 降级需要明确核心链路和非核心链路。
- 兜底结果要可接受，不能产生严重业务误导。
- 熔断和降级都需要监控、告警和恢复策略。
:::

## 3、接口幂等性如何设计
接口幂等性是指同一个请求执行一次和执行多次，最终业务结果一致。它常用于下单、支付回调、消息消费、表单提交等场景。

::: details 详情
### 为什么需要幂等

请求可能因为很多原因重复到达：

- 用户重复点击。
- 网络超时后重试。
- 网关重试。
- 消息队列重复投递。
- 第三方回调重复通知。

如果没有幂等控制，可能出现重复下单、重复扣款、重复发货等问题。

### 幂等键

常见做法是为请求生成业务幂等键：

```txt
userId + orderNo
```

或：

```txt
requestId
```

服务端收到请求后，先检查幂等记录，已经处理过就直接返回之前结果。

### 常见实现方式

- 数据库唯一索引。
- 业务状态机。
- Redis `SET NX`。
- 幂等请求表。
- 分布式锁。
- 消息消费记录表。

### 数据库唯一索引示例

```sql
create unique index uk_order_no on orders(order_no);
```

如果同一个订单号重复插入，数据库会拒绝重复数据。

### 注意事项

- 幂等键要和业务语义绑定。
- 不能只靠前端禁用按钮。
- 重试时要能返回相同业务结果。
- 支付、库存等关键场景要结合事务、锁和状态机。
:::

## 4、缓存和数据库如何保证一致性
缓存和数据库一致性问题通常出现在写入数据库后，缓存中的旧数据仍被读取。实际系统中很难做到绝对强一致，更多是根据业务选择最终一致方案。

::: details 详情
### 常见策略

常见做法包括：

- 先更新数据库，再删除缓存。
- 先删除缓存，再更新数据库。
- 延迟双删。
- 通过消息队列异步删除缓存。
- 订阅数据库变更同步缓存。

### 推荐常见做法

多数业务更常用：

```txt
先更新数据库 -> 再删除缓存
```

下次读取时缓存不存在，再从数据库加载新数据并写回缓存。

### 为什么不是更新缓存

直接更新缓存可能遇到：

- 并发写入导致覆盖。
- 缓存结构和数据库结构不一致。
- 部分缓存组合数据难以准确更新。

删除缓存通常比更新缓存更简单、更稳。

### 延迟双删

延迟双删流程：

```txt
删除缓存 -> 更新数据库 -> 延迟一小段时间 -> 再删除缓存
```

它用于降低并发读写导致旧值回写缓存的概率。

### 注意事项

- 缓存一致性要结合业务容忍度设计。
- 对强一致要求高的业务，不应只依赖缓存方案。
- 删除缓存失败要有重试或补偿机制。
- 缓存要设置过期时间，避免长期脏数据。
:::

## 5、消息队列如何保证消息可靠性
消息队列可靠性通常要从生产者、队列本身和消费者三个环节考虑，目标是尽量避免消息丢失，并能处理重复消息和消费失败。

::: details 详情
### 生产者可靠性

生产者发送消息后，需要确认消息是否成功进入队列。

常见手段：

- 发送确认机制。
- 失败重试。
- 本地消息表。
- 事务消息。

如果业务数据已经写入数据库，但消息发送失败，就需要补偿机制。

### 队列可靠性

队列自身要保证消息不会因为服务异常轻易丢失。

常见手段：

- 消息持久化。
- 主从复制。
- 高可用集群。
- 合理的 ACK 机制。

不同消息队列实现细节不同，要结合具体产品理解。

### 消费者可靠性

消费者处理消息后再确认消费成功。

如果处理失败，不应过早 ACK，否则消息可能丢失。

常见处理：

- 消费失败重试。
- 重试次数限制。
- 死信队列。
- 消费幂等。

### 重复消息

即使可靠性设计充分，也要接受消息可能重复。

所以消费者必须做幂等，例如用：

```txt
messageId
businessId + eventType
```

作为消费记录唯一键。

### 注意事项

- 可靠性越高，系统复杂度和成本通常越高。
- 重试要避免无限重试拖垮系统。
- 死信队列要有人处理和告警。
- 消息顺序、延迟和可靠性之间可能存在取舍。
:::

## 6、分布式链路追踪有什么作用
分布式链路追踪用于跟踪一次请求在多个服务之间的调用路径，帮助排查慢接口、错误来源和服务依赖关系。

::: details 详情
### 为什么需要链路追踪

在单体应用中，一个请求通常只经过一个服务。

在微服务系统中，一个请求可能经过：

```txt
网关 -> 用户服务 -> 订单服务 -> 库存服务 -> 支付服务
```

如果请求变慢或失败，需要知道问题出在哪个服务、哪个接口、哪次调用。

### 核心概念

常见概念包括：

- Trace：一次完整请求链路。
- Span：链路中的一次具体操作或调用。
- TraceId：标识一次完整请求。
- SpanId：标识某个调用片段。
- ParentSpanId：标识父调用关系。

### 日志关联

服务日志中通常会打印 `traceId`：

```txt
traceId=abc123 userId=1 path=/api/orders cost=120ms
```

这样可以把分散在多个服务中的日志串起来。

### 常见用途

- 定位慢请求。
- 分析调用依赖。
- 排查错误传播路径。
- 统计接口耗时。
- 辅助容量评估。

### 注意事项

- TraceId 要在服务间透传。
- 异步任务、消息队列也要传递链路上下文。
- 采样率要平衡成本和可观测性。
- 链路追踪要和日志、指标、告警结合使用。
:::

## 7、高并发读写系统如何设计
高并发读写系统设计要先区分读多写少、写多读少还是读写都高，再围绕缓存、数据库、队列、分片、限流和降级做组合设计。

::: details 详情
### 先分析业务模型

设计前要明确：

- QPS 大概多少。
- 读写比例。
- 数据一致性要求。
- 是否有热点数据。
- 是否允许异步处理。
- 峰值流量来源。

不同业务模型对应不同方案。

### 读多写少

读多写少常见方案：

- CDN。
- 本地缓存。
- Redis 缓存。
- 读写分离。
- 页面静态化。
- 热点数据预热。

重点是降低数据库读压力。

### 写多场景

写多场景常见方案：

- 消息队列削峰。
- 批量写入。
- 分库分表。
- 异步处理。
- 写入限流。
- 幂等控制。

重点是保护数据库和下游服务。

### 热点问题

热点 key、热点商品、热点接口可能导致局部瓶颈。

常见处理：

- 热点缓存。
- 多级缓存。
- 请求合并。
- 分片 key。
- 限流和降级。

### 注意事项

- 高并发不是只加缓存。
- 一致性要求越高，设计越复杂。
- 队列削峰会牺牲实时性。
- 所有方案都要配合监控、压测和容量评估。
:::

## 8、分库分表适合解决什么问题
分库分表用于解决单库或单表在数据量、写入吞吐、存储容量和查询性能上的瓶颈。它不是默认方案，通常是在垂直优化和索引优化后仍无法满足需求时考虑。

::: details 详情
### 分库

分库是把数据拆到多个数据库实例中。

常见目的：

- 分摊存储压力。
- 分摊连接和写入压力。
- 降低单实例故障影响。

### 分表

分表是把一张大表拆成多张小表。

例如订单表：

```txt
orders_00
orders_01
orders_02
...
```

可以按照用户 ID、订单 ID、时间等规则拆分。

### 分片键

分片键选择很关键。

好的分片键应该：

- 查询中经常出现。
- 分布相对均匀。
- 不容易频繁变化。
- 能避免热点集中。

### 带来的问题

分库分表会引入很多复杂度：

- 跨库 join 困难。
- 分布式事务复杂。
- 全局唯一 ID。
- 分页和排序复杂。
- 扩容和迁移成本高。
- 数据聚合查询困难。

### 注意事项

- 分库分表是复杂度很高的方案，不要过早使用。
- 能用索引、缓存、读写分离解决的，不一定要分库分表。
- 分片规则要提前考虑扩容。
- 后台管理、报表和数据分析通常需要额外方案。
:::

## 9、分布式系统如何生成全局唯一 ID
全局唯一 ID 用于在分布式系统中为订单、用户、消息等数据生成唯一标识。常见方案包括 UUID、数据库号段、Redis 自增和雪花算法。

::: details 详情
### UUID

UUID 生成简单，不依赖中心服务。

优点：

- 本地生成。
- 冲突概率极低。

缺点：

- 字符串较长。
- 不递增。
- 作为数据库主键可能影响索引局部性。

### 数据库自增

单库自增 ID 简单可靠，但分布式场景会有单点和性能瓶颈。

可以使用号段模式：

```txt
服务一次申请一段 ID，例如 1000 到 1999
```

服务在本地发号，号段用完后再申请。

### Redis 自增

使用 Redis `INCR` 可以生成递增 ID。

优点是简单，缺点是依赖 Redis 可用性，并需要考虑持久化和高可用。

### 雪花算法

雪花算法通常由：

- 时间戳。
- 机器 ID。
- 序列号。

组合生成 64 位整数。

它可以本地生成，趋势递增，适合高并发场景。

### 注意事项

- ID 是否需要递增会影响方案选择。
- 机器 ID 冲突会导致 ID 冲突。
- 雪花算法要处理时钟回拨。
- 数据库主键要考虑索引性能和存储成本。
:::

## 10、负载均衡常见策略有哪些
负载均衡用于把请求分发到多台后端实例，提升系统吞吐、可用性和扩展能力。常见策略包括轮询、加权轮询、最少连接、哈希和一致性哈希。

::: details 详情
### 常见策略

- 轮询：按顺序把请求分发给后端实例。
- 加权轮询：给性能更强的实例更高权重。
- 最少连接：优先分发给当前连接数更少的实例。
- IP Hash：同一客户端 IP 尽量落到同一实例。
- 一致性哈希：节点增删时尽量减少请求迁移范围。

### 如何选择

普通无状态服务可以使用轮询或加权轮询。

长连接服务更适合关注连接数、连接时长和实例负载。

需要会话粘滞时，可以使用 IP Hash 或基于 Cookie 的路由，但更推荐把会话状态外置到 Redis、数据库或独立 Session 服务中。

### 健康检查

负载均衡不只是分发请求，还要识别不可用实例。

健康检查通常包括：

- TCP 端口是否可连接。
- HTTP 健康检查接口是否返回成功。
- 实例是否处于启动、下线或熔断状态。
- 错误率和响应时间是否异常。

### 注意事项

- 后端服务应尽量无状态，降低路由粘滞依赖。
- 权重调整要结合监控数据，不要凭感觉配置。
- 健康检查过于频繁会增加压力，过于宽松会放大故障。
- 负载均衡层本身也要避免单点故障。
:::

## 11、数据库读写分离如何设计
读写分离是指写请求进入主库，读请求尽量走从库，用来降低主库压力并提升读请求吞吐。它适合读多写少、读流量明显大于写流量的系统。

::: details 详情
### 基本结构

常见结构是：

```txt
业务服务 -> 数据访问层 -> 主库写入
                  -> 从库读取
```

主库负责写入和复制日志，从库通过复制机制同步主库数据。

### 路由规则

读写分离通常在以下位置实现：

- 业务代码中区分读写数据源。
- ORM 或数据库中间件自动路由。
- 数据库代理层统一转发。

写操作、事务内查询和强一致读通常走主库。

普通列表、详情页、统计类读请求可以走从库。

### 主从延迟

读写分离最大的问题是主从复制延迟。

例如用户刚修改资料，马上读取详情，如果读到从库，可能看到旧数据。

常见解决方式：

- 写后短时间内读主库。
- 关键业务强制读主库。
- 根据复制延迟动态摘除落后从库。
- 前端或服务端做短暂状态补偿。

### 注意事项

- 读写分离不能提升写入能力，写瓶颈仍在主库。
- 从库数量越多，复制和运维复杂度越高。
- 事务一致性要求高的场景要谨慎读从库。
- 需要监控复制延迟、主从状态和从库错误率。
:::

## 12、系统中的热点数据如何治理
热点数据是指短时间内被大量访问或更新的数据，例如热门商品、爆款活动、明星动态和秒杀库存。治理热点的核心是识别热点、分散压力、保护后端。

::: details 详情
### 热点类型

常见热点包括：

- 热点读：大量请求读取同一份数据。
- 热点写：大量请求更新同一个 key 或同一行记录。
- 热点缓存 key：大量流量集中到一个缓存 key。
- 热点分片：分库分表后某个分片压力明显更高。

### 热点读治理

热点读可以通过缓存和多级缓存缓解：

- CDN 缓存静态内容。
- 本地缓存减少远程缓存访问。
- Redis 缓存热点数据。
- 热点 key 副本拆分。
- 提前预热缓存。

对读多写少的数据，可以适当延长缓存时间。

### 热点写治理

热点写更难处理，因为写请求往往需要一致性。

常见方式：

- 请求排队削峰。
- 分桶计数后异步汇总。
- 使用消息队列串行化处理。
- 将强一致写改为最终一致。
- 对超出容量的请求做限流或降级。

### 识别热点

热点治理依赖监控：

- 接口 QPS。
- 缓存 key 访问频率。
- 数据库行锁等待。
- 分片负载。
- 错误率和响应时间。

### 注意事项

- 不要只靠扩容解决热点写，单点更新仍可能成为瓶颈。
- 热点 key 失效要防止缓存击穿。
- 秒杀类场景要把限流、库存预扣和队列一起设计。
- 热点规则要动态化，避免只依赖人工配置。
:::

## 13、服务调用超时应该如何设计
服务调用超时用于限制一次远程调用最多等待多久，避免上游线程、连接和请求队列被慢请求拖垮。合理的超时设计是系统稳定性的基础。

::: details 详情
### 为什么必须设置超时

如果没有超时，一个下游服务变慢可能导致：

- 上游请求持续等待。
- 线程池被占满。
- 连接池耗尽。
- 队列堆积。
- 故障沿调用链扩散。

超时的目标不是让请求一定成功，而是在不可控等待前尽快释放资源。

### 如何设置超时时间

超时时间要结合业务和链路延迟设置。

可以参考：

- 下游服务 P95、P99 响应时间。
- 当前接口的整体 SLA。
- 调用链中剩余时间预算。
- 是否允许重试。
- 用户是否正在等待结果。

例如一个接口整体要求 1 秒内返回，就不能给某个下游调用单独设置 2 秒超时。

### 超时后的处理

超时后常见处理方式：

- 返回明确错误。
- 走降级数据。
- 使用缓存兜底。
- 异步补偿。
- 触发熔断或限流。

### 注意事项

- 超时要和重试一起设计，避免重试把压力放大。
- 客户端、网关、服务端和数据库都要有超时配置。
- 长任务不适合同步等待，应改为异步任务和状态查询。
- 超时日志要包含下游服务、耗时、请求 ID 和错误类型，方便排查。
:::

## 14、强一致性和最终一致性有什么区别
强一致性要求写入成功后，后续读取必须读到最新结果；最终一致性允许短时间内读到旧数据，但系统会在一段时间后收敛到一致状态。

::: details 详情
### 强一致性

强一致性更适合对正确性要求极高的场景。

例如：

- 支付扣款。
- 库存扣减。
- 账户余额。
- 权限变更。

它的优点是语义清晰，缺点是系统复杂度、延迟和可用性压力更高。

### 最终一致性

最终一致性更适合允许短暂延迟的场景。

例如：

- 订单状态同步到搜索索引。
- 用户操作日志。
- 统计报表。
- 消息通知。
- 缓存更新。

常见实现方式包括消息队列、异步任务、重试补偿和定时对账。

### 如何选择

选择一致性模型要看业务后果。

如果读到旧数据会导致资金损失、越权或重复发货，应优先强一致。

如果读到旧数据只影响展示体验，可以考虑最终一致，并通过提示、刷新和补偿机制降低影响。

### 注意事项

- 最终一致不是不一致，需要有补偿和对账机制。
- 强一致通常会牺牲一部分性能或可用性。
- 用户感知强的场景可以用“写后读主库”提升体验。
- 设计接口时要明确一致性承诺，避免调用方误用。
:::

## 15、CAP 理论是什么
CAP 理论用于描述分布式系统在一致性、可用性和分区容错性之间的取舍。网络分区发生时，系统通常无法同时保证强一致性和高可用性。

::: details 详情
### 三个概念

- C：Consistency，一致性，所有节点读取到的数据一致。
- A：Availability，可用性，每个请求都能得到非错误响应。
- P：Partition Tolerance，分区容错性，网络分区时系统仍能继续运行。

在分布式系统中，网络分区无法完全避免，因此 P 通常必须考虑。

### CP 系统

CP 更偏向一致性。

当网络分区发生时，为了避免读写到不一致数据，部分节点可能拒绝服务。

适合对正确性要求高的场景，例如配置中心、分布式锁、账务核心数据。

### AP 系统

AP 更偏向可用性。

当网络分区发生时，系统尽量继续响应请求，但不同节点之间可能短时间数据不一致。

适合允许最终一致的场景，例如动态、点赞、消息通知、缓存同步。

### 注意事项

- CAP 不是说平时只能三选二，而是网络分区发生时要做取舍。
- 一致性也有强一致、顺序一致、最终一致等不同层次。
- 具体系统常在不同业务链路上做不同取舍。
- 面试回答要结合业务场景，不要只背定义。
:::

## 16、BASE 理论是什么
BASE 理论是分布式系统中对强一致性的放松，强调基本可用、软状态和最终一致性。它常用于高可用、高并发但允许短暂不一致的业务场景。

::: details 详情
### 三个含义

BASE 包括：

- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

它不是放弃一致性，而是接受系统在短时间内处于中间状态，最终通过同步、补偿和对账达到一致。

### 基本可用

基本可用是指故障或高峰时系统仍能提供核心能力。

例如：

- 非核心功能降级。
- 页面展示缓存数据。
- 限制部分用户访问。
- 排队处理请求。

### 软状态

软状态表示系统允许存在中间状态。

例如订单已创建但支付状态还在同步中，搜索索引还没更新到最新数据。

### 最终一致

最终一致表示经过一段时间后，多个副本或多个系统中的数据会收敛到一致。

常见手段：

- 消息队列。
- 定时任务。
- 重试补偿。
- 对账修复。

### 注意事项

- BASE 适合允许短暂不一致的业务，不适合所有场景。
- 要明确用户能看到哪些中间状态。
- 必须设计补偿、重试和对账机制。
- 最终一致要有时间边界和监控，不是无限期等待。
:::

## 17、系统降级和兜底方案如何设计
系统降级是在压力过大、依赖故障或部分能力不可用时，主动关闭或弱化非核心功能，保证核心链路继续可用。兜底方案用于在失败时提供可接受的替代结果。

::: details 详情
### 降级对象

常见降级对象包括：

- 推荐系统。
- 排行榜。
- 评论区。
- 非核心活动位。
- 第三方服务。
- 复杂查询和统计。

核心交易、支付、鉴权等链路通常不能随意降级，需要更严格的容灾方案。

### 兜底方式

常见兜底方式：

- 返回缓存数据。
- 返回默认配置。
- 使用静态文案。
- 隐藏非核心模块。
- 展示稍后重试。
- 切换备用服务。

例如推荐接口失败时，可以返回热门榜单或默认内容。

### 触发条件

降级可以由多种条件触发：

- 错误率升高。
- 响应时间超过阈值。
- 下游服务不可用。
- 线程池或连接池耗尽。
- 人工开关。
- 灰度规则。

### 注意事项

- 降级开关要支持快速生效和回滚。
- 降级结果要对用户可理解，不要静默失败。
- 要记录降级日志和指标，方便复盘。
- 降级不能掩盖长期容量问题，事后仍要修复根因。
:::

## 18、分库分表后如何做数据迁移
分库分表后的数据迁移通常涉及历史数据搬迁、增量数据同步、双写校验和流量切换。目标是在尽量不停机的情况下完成平滑迁移。

::: details 详情
### 常见步骤

一般可以分为：

- 设计新的分片规则。
- 准备目标库表。
- 全量迁移历史数据。
- 同步增量变更。
- 做数据校验。
- 灰度切读流量。
- 切写流量。
- 保留回滚窗口。

迁移过程要尽量可暂停、可重试、可回滚。

### 全量和增量

全量迁移用于搬迁已有数据。

增量同步用于处理迁移期间新产生或变更的数据。

增量同步常见方式：

- 业务双写。
- 读取 binlog。
- 消息队列同步。
- 定时任务补偿。

### 数据校验

迁移后要校验：

- 行数是否一致。
- 主键范围是否完整。
- 核心字段 hash 是否一致。
- 抽样业务查询是否正确。
- 增量延迟是否在可接受范围内。

### 注意事项

- 分片键一旦确定，后续调整成本很高。
- 双写期间要处理部分成功和重试幂等。
- 切流量应灰度进行，不要一次性全量切换。
- 迁移期间要监控延迟、错误率、数据库负载和数据差异。
:::

## 19、接口版本管理如何设计
接口版本管理用于在业务演进时兼容旧客户端和旧调用方，避免服务端变更直接破坏线上功能。它常见于 App、小程序、开放平台和多团队协作系统。

::: details 详情
### 为什么需要版本管理

接口变更可能包括：

- 字段新增。
- 字段删除。
- 字段含义变化。
- 参数格式变化。
- 错误码变化。
- 业务流程变化。

其中新增字段通常兼容性较好，删除字段和改变字段含义风险更高。

### 常见方式

常见版本设计：

- URL 路径版本：`/api/v1/users`。
- Header 版本：`Accept-Version: v1`。
- 查询参数版本：`?version=1`。
- 客户端版本配合灰度策略。

内部系统可以更轻量，开放平台通常要更严格。

### 兼容策略

接口升级时可以：

- 保持旧字段一段时间。
- 新增字段而不是改旧字段含义。
- 服务端根据版本返回不同结构。
- 提供迁移文档和废弃时间表。
- 对旧版本调用量做监控。

### 注意事项

- 不要频繁创建大版本，否则维护成本会急剧上升。
- 破坏性变更必须明确通知调用方。
- 移动端发版慢，要保留足够兼容窗口。
- 错误码、分页结构、时间格式也属于接口契约的一部分。
:::
