# 系统设计

这里用于整理系统设计相关面试题，后续按限流、熔断、降级、缓存一致性、分布式锁、消息队列、幂等、链路追踪、高并发读写等方向追加。

## 1、系统为什么需要限流，常见限流算法有哪些
限流用于限制单位时间内的请求数量，防止流量突增打垮服务。它常用于接口保护、登录防刷、短信发送、支付下单、网关入口等场景。

::: details 详情
### 为什么需要限流

如果没有限流，突发流量可能导致：

- CPU 或内存打满。
- 数据库连接池耗尽。
- 下游服务被压垮。
- 队列积压。
- 正常用户请求被拖慢。

限流的目标不是让所有请求都成功，而是在系统承载范围内保护核心链路。

### 固定窗口算法

把时间按固定窗口切分，例如每分钟最多 100 次。

优点：实现简单。

缺点：窗口边界可能出现流量突刺。

### 滑动窗口算法

记录更细粒度的时间片或请求时间，按最近一段时间统计请求数。

优点：比固定窗口更平滑。

缺点：实现和存储成本更高。

### 漏桶算法

请求进入桶中，按照固定速率流出。

优点：输出流量稳定。

缺点：突发流量处理能力较弱。

### 令牌桶算法

系统按固定速率生成令牌，请求必须拿到令牌才能通过。

优点：既能限制平均速率，也允许一定突发流量。

### 面试要点

- 限流可以放在网关、服务端、用户维度、IP 维度或业务维度。
- 高并发分布式场景常结合 Redis 或网关限流。
- 限流后要返回明确错误码，并配合降级、排队或重试提示。
- 限流阈值需要结合容量压测和业务优先级确定。
:::

## 2、熔断和降级有什么区别
熔断和降级都是系统稳定性手段。熔断更关注在下游异常时快速切断调用，避免故障扩散；降级更关注在资源不足或功能异常时牺牲非核心能力，保证核心链路可用。

::: details 详情
### 熔断

熔断类似电路保险丝。

当某个下游服务错误率或超时率过高时，上游不再继续请求它，而是直接返回失败或兜底结果。

常见状态：

- closed：正常调用。
- open：熔断打开，直接拒绝或兜底。
- half-open：放少量请求探测下游是否恢复。

### 降级

降级是主动降低服务能力，例如：

- 首页不展示推荐模块。
- 搜索只返回缓存结果。
- 评论区暂时关闭。
- 非核心接口返回默认值。
- 暂停低优先级任务。

降级可以是自动触发，也可以由运营或研发手动开关触发。

### 二者区别

- 熔断通常针对下游依赖异常。
- 降级通常针对整体资源压力或非核心能力取舍。
- 熔断强调保护调用链。
- 降级强调保核心、舍非核心。

### 组合使用

实际系统中二者经常一起使用：

```txt
下游推荐服务超时
-> 熔断推荐服务调用
-> 降级为默认推荐列表
```

### 面试要点

- 熔断需要指标：错误率、超时率、并发数等。
- 降级需要明确核心链路和非核心链路。
- 兜底结果要可接受，不能产生严重业务误导。
- 熔断和降级都需要监控、告警和恢复策略。
:::

## 3、接口幂等性如何设计
接口幂等性是指同一个请求执行一次和执行多次，最终业务结果一致。它常用于下单、支付回调、消息消费、表单提交等场景。

::: details 详情
### 为什么需要幂等

请求可能因为很多原因重复到达：

- 用户重复点击。
- 网络超时后重试。
- 网关重试。
- 消息队列重复投递。
- 第三方回调重复通知。

如果没有幂等控制，可能出现重复下单、重复扣款、重复发货等问题。

### 幂等键

常见做法是为请求生成业务幂等键：

```txt
userId + orderNo
```

或：

```txt
requestId
```

服务端收到请求后，先检查幂等记录，已经处理过就直接返回之前结果。

### 常见实现方式

- 数据库唯一索引。
- 业务状态机。
- Redis `SET NX`。
- 幂等请求表。
- 分布式锁。
- 消息消费记录表。

### 数据库唯一索引示例

```sql
create unique index uk_order_no on orders(order_no);
```

如果同一个订单号重复插入，数据库会拒绝重复数据。

### 注意事项

- 幂等键要和业务语义绑定。
- 不能只靠前端禁用按钮。
- 重试时要能返回相同业务结果。
- 支付、库存等关键场景要结合事务、锁和状态机。
:::

## 4、缓存和数据库如何保证一致性
缓存和数据库一致性问题通常出现在写入数据库后，缓存中的旧数据仍被读取。实际系统中很难做到绝对强一致，更多是根据业务选择最终一致方案。

::: details 详情
### 常见策略

常见做法包括：

- 先更新数据库，再删除缓存。
- 先删除缓存，再更新数据库。
- 延迟双删。
- 通过消息队列异步删除缓存。
- 订阅数据库变更同步缓存。

### 推荐常见做法

多数业务更常用：

```txt
先更新数据库 -> 再删除缓存
```

下次读取时缓存不存在，再从数据库加载新数据并写回缓存。

### 为什么不是更新缓存

直接更新缓存可能遇到：

- 并发写入导致覆盖。
- 缓存结构和数据库结构不一致。
- 部分缓存组合数据难以准确更新。

删除缓存通常比更新缓存更简单、更稳。

### 延迟双删

延迟双删流程：

```txt
删除缓存 -> 更新数据库 -> 延迟一小段时间 -> 再删除缓存
```

它用于降低并发读写导致旧值回写缓存的概率。

### 注意事项

- 缓存一致性要结合业务容忍度设计。
- 对强一致要求高的业务，不应只依赖缓存方案。
- 删除缓存失败要有重试或补偿机制。
- 缓存要设置过期时间，避免长期脏数据。
:::

## 5、消息队列如何保证消息可靠性
消息队列可靠性通常要从生产者、队列本身和消费者三个环节考虑，目标是尽量避免消息丢失，并能处理重复消息和消费失败。

::: details 详情
### 生产者可靠性

生产者发送消息后，需要确认消息是否成功进入队列。

常见手段：

- 发送确认机制。
- 失败重试。
- 本地消息表。
- 事务消息。

如果业务数据已经写入数据库，但消息发送失败，就需要补偿机制。

### 队列可靠性

队列自身要保证消息不会因为服务异常轻易丢失。

常见手段：

- 消息持久化。
- 主从复制。
- 高可用集群。
- 合理的 ACK 机制。

不同消息队列实现细节不同，要结合具体产品理解。

### 消费者可靠性

消费者处理消息后再确认消费成功。

如果处理失败，不应过早 ACK，否则消息可能丢失。

常见处理：

- 消费失败重试。
- 重试次数限制。
- 死信队列。
- 消费幂等。

### 重复消息

即使可靠性设计充分，也要接受消息可能重复。

所以消费者必须做幂等，例如用：

```txt
messageId
businessId + eventType
```

作为消费记录唯一键。

### 注意事项

- 可靠性越高，系统复杂度和成本通常越高。
- 重试要避免无限重试拖垮系统。
- 死信队列要有人处理和告警。
- 消息顺序、延迟和可靠性之间可能存在取舍。
:::

## 6、分布式链路追踪有什么作用
分布式链路追踪用于跟踪一次请求在多个服务之间的调用路径，帮助排查慢接口、错误来源和服务依赖关系。

::: details 详情
### 为什么需要链路追踪

在单体应用中，一个请求通常只经过一个服务。

在微服务系统中，一个请求可能经过：

```txt
网关 -> 用户服务 -> 订单服务 -> 库存服务 -> 支付服务
```

如果请求变慢或失败，需要知道问题出在哪个服务、哪个接口、哪次调用。

### 核心概念

常见概念包括：

- Trace：一次完整请求链路。
- Span：链路中的一次具体操作或调用。
- TraceId：标识一次完整请求。
- SpanId：标识某个调用片段。
- ParentSpanId：标识父调用关系。

### 日志关联

服务日志中通常会打印 `traceId`：

```txt
traceId=abc123 userId=1 path=/api/orders cost=120ms
```

这样可以把分散在多个服务中的日志串起来。

### 常见用途

- 定位慢请求。
- 分析调用依赖。
- 排查错误传播路径。
- 统计接口耗时。
- 辅助容量评估。

### 注意事项

- TraceId 要在服务间透传。
- 异步任务、消息队列也要传递链路上下文。
- 采样率要平衡成本和可观测性。
- 链路追踪要和日志、指标、告警结合使用。
:::
