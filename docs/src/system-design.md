# 系统设计

这里用于整理系统设计相关面试题，后续按限流、熔断、降级、缓存一致性、分布式锁、消息队列、幂等、链路追踪、高并发读写等方向追加。

## 1、系统为什么需要限流，常见限流算法有哪些
限流用于限制单位时间内的请求数量，防止流量突增打垮服务。它常用于接口保护、登录防刷、短信发送、支付下单、网关入口等场景。

::: details 详情
### 为什么需要限流

如果没有限流，突发流量可能导致：

- CPU 或内存打满。
- 数据库连接池耗尽。
- 下游服务被压垮。
- 队列积压。
- 正常用户请求被拖慢。

限流的目标不是让所有请求都成功，而是在系统承载范围内保护核心链路。

### 固定窗口算法

把时间按固定窗口切分，例如每分钟最多 100 次。

优点：实现简单。

缺点：窗口边界可能出现流量突刺。

### 滑动窗口算法

记录更细粒度的时间片或请求时间，按最近一段时间统计请求数。

优点：比固定窗口更平滑。

缺点：实现和存储成本更高。

### 漏桶算法

请求进入桶中，按照固定速率流出。

优点：输出流量稳定。

缺点：突发流量处理能力较弱。

### 令牌桶算法

系统按固定速率生成令牌，请求必须拿到令牌才能通过。

优点：既能限制平均速率，也允许一定突发流量。

### 面试要点

- 限流可以放在网关、服务端、用户维度、IP 维度或业务维度。
- 高并发分布式场景常结合 Redis 或网关限流。
- 限流后要返回明确错误码，并配合降级、排队或重试提示。
- 限流阈值需要结合容量压测和业务优先级确定。
:::

## 2、熔断和降级有什么区别
熔断和降级都是系统稳定性手段。熔断更关注在下游异常时快速切断调用，避免故障扩散；降级更关注在资源不足或功能异常时牺牲非核心能力，保证核心链路可用。

::: details 详情
### 熔断

熔断类似电路保险丝。

当某个下游服务错误率或超时率过高时，上游不再继续请求它，而是直接返回失败或兜底结果。

常见状态：

- closed：正常调用。
- open：熔断打开，直接拒绝或兜底。
- half-open：放少量请求探测下游是否恢复。

### 降级

降级是主动降低服务能力，例如：

- 首页不展示推荐模块。
- 搜索只返回缓存结果。
- 评论区暂时关闭。
- 非核心接口返回默认值。
- 暂停低优先级任务。

降级可以是自动触发，也可以由运营或研发手动开关触发。

### 二者区别

- 熔断通常针对下游依赖异常。
- 降级通常针对整体资源压力或非核心能力取舍。
- 熔断强调保护调用链。
- 降级强调保核心、舍非核心。

### 组合使用

实际系统中二者经常一起使用：

```txt
下游推荐服务超时
-> 熔断推荐服务调用
-> 降级为默认推荐列表
```

### 面试要点

- 熔断需要指标：错误率、超时率、并发数等。
- 降级需要明确核心链路和非核心链路。
- 兜底结果要可接受，不能产生严重业务误导。
- 熔断和降级都需要监控、告警和恢复策略。
:::

## 3、接口幂等性如何设计
接口幂等性是指同一个请求执行一次和执行多次，最终业务结果一致。它常用于下单、支付回调、消息消费、表单提交等场景。

::: details 详情
### 为什么需要幂等

请求可能因为很多原因重复到达：

- 用户重复点击。
- 网络超时后重试。
- 网关重试。
- 消息队列重复投递。
- 第三方回调重复通知。

如果没有幂等控制，可能出现重复下单、重复扣款、重复发货等问题。

### 幂等键

常见做法是为请求生成业务幂等键：

```txt
userId + orderNo
```

或：

```txt
requestId
```

服务端收到请求后，先检查幂等记录，已经处理过就直接返回之前结果。

### 常见实现方式

- 数据库唯一索引。
- 业务状态机。
- Redis `SET NX`。
- 幂等请求表。
- 分布式锁。
- 消息消费记录表。

### 数据库唯一索引示例

```sql
create unique index uk_order_no on orders(order_no);
```

如果同一个订单号重复插入，数据库会拒绝重复数据。

### 注意事项

- 幂等键要和业务语义绑定。
- 不能只靠前端禁用按钮。
- 重试时要能返回相同业务结果。
- 支付、库存等关键场景要结合事务、锁和状态机。
:::

## 4、缓存和数据库如何保证一致性
缓存和数据库一致性问题通常出现在写入数据库后，缓存中的旧数据仍被读取。实际系统中很难做到绝对强一致，更多是根据业务选择最终一致方案。

::: details 详情
### 常见策略

常见做法包括：

- 先更新数据库，再删除缓存。
- 先删除缓存，再更新数据库。
- 延迟双删。
- 通过消息队列异步删除缓存。
- 订阅数据库变更同步缓存。

### 推荐常见做法

多数业务更常用：

```txt
先更新数据库 -> 再删除缓存
```

下次读取时缓存不存在，再从数据库加载新数据并写回缓存。

### 为什么不是更新缓存

直接更新缓存可能遇到：

- 并发写入导致覆盖。
- 缓存结构和数据库结构不一致。
- 部分缓存组合数据难以准确更新。

删除缓存通常比更新缓存更简单、更稳。

### 延迟双删

延迟双删流程：

```txt
删除缓存 -> 更新数据库 -> 延迟一小段时间 -> 再删除缓存
```

它用于降低并发读写导致旧值回写缓存的概率。

### 注意事项

- 缓存一致性要结合业务容忍度设计。
- 对强一致要求高的业务，不应只依赖缓存方案。
- 删除缓存失败要有重试或补偿机制。
- 缓存要设置过期时间，避免长期脏数据。
:::

## 5、消息队列如何保证消息可靠性
消息队列可靠性通常要从生产者、队列本身和消费者三个环节考虑，目标是尽量避免消息丢失，并能处理重复消息和消费失败。

::: details 详情
### 生产者可靠性

生产者发送消息后，需要确认消息是否成功进入队列。

常见手段：

- 发送确认机制。
- 失败重试。
- 本地消息表。
- 事务消息。

如果业务数据已经写入数据库，但消息发送失败，就需要补偿机制。

### 队列可靠性

队列自身要保证消息不会因为服务异常轻易丢失。

常见手段：

- 消息持久化。
- 主从复制。
- 高可用集群。
- 合理的 ACK 机制。

不同消息队列实现细节不同，要结合具体产品理解。

### 消费者可靠性

消费者处理消息后再确认消费成功。

如果处理失败，不应过早 ACK，否则消息可能丢失。

常见处理：

- 消费失败重试。
- 重试次数限制。
- 死信队列。
- 消费幂等。

### 重复消息

即使可靠性设计充分，也要接受消息可能重复。

所以消费者必须做幂等，例如用：

```txt
messageId
businessId + eventType
```

作为消费记录唯一键。

### 注意事项

- 可靠性越高，系统复杂度和成本通常越高。
- 重试要避免无限重试拖垮系统。
- 死信队列要有人处理和告警。
- 消息顺序、延迟和可靠性之间可能存在取舍。
:::

## 6、分布式链路追踪有什么作用
分布式链路追踪用于跟踪一次请求在多个服务之间的调用路径，帮助排查慢接口、错误来源和服务依赖关系。

::: details 详情
### 为什么需要链路追踪

在单体应用中，一个请求通常只经过一个服务。

在微服务系统中，一个请求可能经过：

```txt
网关 -> 用户服务 -> 订单服务 -> 库存服务 -> 支付服务
```

如果请求变慢或失败，需要知道问题出在哪个服务、哪个接口、哪次调用。

### 核心概念

常见概念包括：

- Trace：一次完整请求链路。
- Span：链路中的一次具体操作或调用。
- TraceId：标识一次完整请求。
- SpanId：标识某个调用片段。
- ParentSpanId：标识父调用关系。

### 日志关联

服务日志中通常会打印 `traceId`：

```txt
traceId=abc123 userId=1 path=/api/orders cost=120ms
```

这样可以把分散在多个服务中的日志串起来。

### 常见用途

- 定位慢请求。
- 分析调用依赖。
- 排查错误传播路径。
- 统计接口耗时。
- 辅助容量评估。

### 注意事项

- TraceId 要在服务间透传。
- 异步任务、消息队列也要传递链路上下文。
- 采样率要平衡成本和可观测性。
- 链路追踪要和日志、指标、告警结合使用。
:::

## 7、高并发读写系统如何设计
高并发读写系统设计要先区分读多写少、写多读少还是读写都高，再围绕缓存、数据库、队列、分片、限流和降级做组合设计。

::: details 详情
### 先分析业务模型

设计前要明确：

- QPS 大概多少。
- 读写比例。
- 数据一致性要求。
- 是否有热点数据。
- 是否允许异步处理。
- 峰值流量来源。

不同业务模型对应不同方案。

### 读多写少

读多写少常见方案：

- CDN。
- 本地缓存。
- Redis 缓存。
- 读写分离。
- 页面静态化。
- 热点数据预热。

重点是降低数据库读压力。

### 写多场景

写多场景常见方案：

- 消息队列削峰。
- 批量写入。
- 分库分表。
- 异步处理。
- 写入限流。
- 幂等控制。

重点是保护数据库和下游服务。

### 热点问题

热点 key、热点商品、热点接口可能导致局部瓶颈。

常见处理：

- 热点缓存。
- 多级缓存。
- 请求合并。
- 分片 key。
- 限流和降级。

### 注意事项

- 高并发不是只加缓存。
- 一致性要求越高，设计越复杂。
- 队列削峰会牺牲实时性。
- 所有方案都要配合监控、压测和容量评估。
:::

## 8、分库分表适合解决什么问题
分库分表用于解决单库或单表在数据量、写入吞吐、存储容量和查询性能上的瓶颈。它不是默认方案，通常是在垂直优化和索引优化后仍无法满足需求时考虑。

::: details 详情
### 分库

分库是把数据拆到多个数据库实例中。

常见目的：

- 分摊存储压力。
- 分摊连接和写入压力。
- 降低单实例故障影响。

### 分表

分表是把一张大表拆成多张小表。

例如订单表：

```txt
orders_00
orders_01
orders_02
...
```

可以按照用户 ID、订单 ID、时间等规则拆分。

### 分片键

分片键选择很关键。

好的分片键应该：

- 查询中经常出现。
- 分布相对均匀。
- 不容易频繁变化。
- 能避免热点集中。

### 带来的问题

分库分表会引入很多复杂度：

- 跨库 join 困难。
- 分布式事务复杂。
- 全局唯一 ID。
- 分页和排序复杂。
- 扩容和迁移成本高。
- 数据聚合查询困难。

### 注意事项

- 分库分表是复杂度很高的方案，不要过早使用。
- 能用索引、缓存、读写分离解决的，不一定要分库分表。
- 分片规则要提前考虑扩容。
- 后台管理、报表和数据分析通常需要额外方案。
:::

## 9、分布式系统如何生成全局唯一 ID
全局唯一 ID 用于在分布式系统中为订单、用户、消息等数据生成唯一标识。常见方案包括 UUID、数据库号段、Redis 自增和雪花算法。

::: details 详情
### UUID

UUID 生成简单，不依赖中心服务。

优点：

- 本地生成。
- 冲突概率极低。

缺点：

- 字符串较长。
- 不递增。
- 作为数据库主键可能影响索引局部性。

### 数据库自增

单库自增 ID 简单可靠，但分布式场景会有单点和性能瓶颈。

可以使用号段模式：

```txt
服务一次申请一段 ID，例如 1000 到 1999
```

服务在本地发号，号段用完后再申请。

### Redis 自增

使用 Redis `INCR` 可以生成递增 ID。

优点是简单，缺点是依赖 Redis 可用性，并需要考虑持久化和高可用。

### 雪花算法

雪花算法通常由：

- 时间戳。
- 机器 ID。
- 序列号。

组合生成 64 位整数。

它可以本地生成，趋势递增，适合高并发场景。

### 注意事项

- ID 是否需要递增会影响方案选择。
- 机器 ID 冲突会导致 ID 冲突。
- 雪花算法要处理时钟回拨。
- 数据库主键要考虑索引性能和存储成本。
:::

## 10、负载均衡常见策略有哪些
负载均衡用于把请求分发到多台后端实例，提升系统吞吐、可用性和扩展能力。常见策略包括轮询、加权轮询、最少连接、哈希和一致性哈希。

::: details 详情
### 常见策略

- 轮询：按顺序把请求分发给后端实例。
- 加权轮询：给性能更强的实例更高权重。
- 最少连接：优先分发给当前连接数更少的实例。
- IP Hash：同一客户端 IP 尽量落到同一实例。
- 一致性哈希：节点增删时尽量减少请求迁移范围。

### 如何选择

普通无状态服务可以使用轮询或加权轮询。

长连接服务更适合关注连接数、连接时长和实例负载。

需要会话粘滞时，可以使用 IP Hash 或基于 Cookie 的路由，但更推荐把会话状态外置到 Redis、数据库或独立 Session 服务中。

### 健康检查

负载均衡不只是分发请求，还要识别不可用实例。

健康检查通常包括：

- TCP 端口是否可连接。
- HTTP 健康检查接口是否返回成功。
- 实例是否处于启动、下线或熔断状态。
- 错误率和响应时间是否异常。

### 注意事项

- 后端服务应尽量无状态，降低路由粘滞依赖。
- 权重调整要结合监控数据，不要凭感觉配置。
- 健康检查过于频繁会增加压力，过于宽松会放大故障。
- 负载均衡层本身也要避免单点故障。
:::

## 11、数据库读写分离如何设计
读写分离是指写请求进入主库，读请求尽量走从库，用来降低主库压力并提升读请求吞吐。它适合读多写少、读流量明显大于写流量的系统。

::: details 详情
### 基本结构

常见结构是：

```txt
业务服务 -> 数据访问层 -> 主库写入
                  -> 从库读取
```

主库负责写入和复制日志，从库通过复制机制同步主库数据。

### 路由规则

读写分离通常在以下位置实现：

- 业务代码中区分读写数据源。
- ORM 或数据库中间件自动路由。
- 数据库代理层统一转发。

写操作、事务内查询和强一致读通常走主库。

普通列表、详情页、统计类读请求可以走从库。

### 主从延迟

读写分离最大的问题是主从复制延迟。

例如用户刚修改资料，马上读取详情，如果读到从库，可能看到旧数据。

常见解决方式：

- 写后短时间内读主库。
- 关键业务强制读主库。
- 根据复制延迟动态摘除落后从库。
- 前端或服务端做短暂状态补偿。

### 注意事项

- 读写分离不能提升写入能力，写瓶颈仍在主库。
- 从库数量越多，复制和运维复杂度越高。
- 事务一致性要求高的场景要谨慎读从库。
- 需要监控复制延迟、主从状态和从库错误率。
:::

## 12、系统中的热点数据如何治理
热点数据是指短时间内被大量访问或更新的数据，例如热门商品、爆款活动、明星动态和秒杀库存。治理热点的核心是识别热点、分散压力、保护后端。

::: details 详情
### 热点类型

常见热点包括：

- 热点读：大量请求读取同一份数据。
- 热点写：大量请求更新同一个 key 或同一行记录。
- 热点缓存 key：大量流量集中到一个缓存 key。
- 热点分片：分库分表后某个分片压力明显更高。

### 热点读治理

热点读可以通过缓存和多级缓存缓解：

- CDN 缓存静态内容。
- 本地缓存减少远程缓存访问。
- Redis 缓存热点数据。
- 热点 key 副本拆分。
- 提前预热缓存。

对读多写少的数据，可以适当延长缓存时间。

### 热点写治理

热点写更难处理，因为写请求往往需要一致性。

常见方式：

- 请求排队削峰。
- 分桶计数后异步汇总。
- 使用消息队列串行化处理。
- 将强一致写改为最终一致。
- 对超出容量的请求做限流或降级。

### 识别热点

热点治理依赖监控：

- 接口 QPS。
- 缓存 key 访问频率。
- 数据库行锁等待。
- 分片负载。
- 错误率和响应时间。

### 注意事项

- 不要只靠扩容解决热点写，单点更新仍可能成为瓶颈。
- 热点 key 失效要防止缓存击穿。
- 秒杀类场景要把限流、库存预扣和队列一起设计。
- 热点规则要动态化，避免只依赖人工配置。
:::

## 13、服务调用超时应该如何设计
服务调用超时用于限制一次远程调用最多等待多久，避免上游线程、连接和请求队列被慢请求拖垮。合理的超时设计是系统稳定性的基础。

::: details 详情
### 为什么必须设置超时

如果没有超时，一个下游服务变慢可能导致：

- 上游请求持续等待。
- 线程池被占满。
- 连接池耗尽。
- 队列堆积。
- 故障沿调用链扩散。

超时的目标不是让请求一定成功，而是在不可控等待前尽快释放资源。

### 如何设置超时时间

超时时间要结合业务和链路延迟设置。

可以参考：

- 下游服务 P95、P99 响应时间。
- 当前接口的整体 SLA。
- 调用链中剩余时间预算。
- 是否允许重试。
- 用户是否正在等待结果。

例如一个接口整体要求 1 秒内返回，就不能给某个下游调用单独设置 2 秒超时。

### 超时后的处理

超时后常见处理方式：

- 返回明确错误。
- 走降级数据。
- 使用缓存兜底。
- 异步补偿。
- 触发熔断或限流。

### 注意事项

- 超时要和重试一起设计，避免重试把压力放大。
- 客户端、网关、服务端和数据库都要有超时配置。
- 长任务不适合同步等待，应改为异步任务和状态查询。
- 超时日志要包含下游服务、耗时、请求 ID 和错误类型，方便排查。
:::

## 14、强一致性和最终一致性有什么区别
强一致性要求写入成功后，后续读取必须读到最新结果；最终一致性允许短时间内读到旧数据，但系统会在一段时间后收敛到一致状态。

::: details 详情
### 强一致性

强一致性更适合对正确性要求极高的场景。

例如：

- 支付扣款。
- 库存扣减。
- 账户余额。
- 权限变更。

它的优点是语义清晰，缺点是系统复杂度、延迟和可用性压力更高。

### 最终一致性

最终一致性更适合允许短暂延迟的场景。

例如：

- 订单状态同步到搜索索引。
- 用户操作日志。
- 统计报表。
- 消息通知。
- 缓存更新。

常见实现方式包括消息队列、异步任务、重试补偿和定时对账。

### 如何选择

选择一致性模型要看业务后果。

如果读到旧数据会导致资金损失、越权或重复发货，应优先强一致。

如果读到旧数据只影响展示体验，可以考虑最终一致，并通过提示、刷新和补偿机制降低影响。

### 注意事项

- 最终一致不是不一致，需要有补偿和对账机制。
- 强一致通常会牺牲一部分性能或可用性。
- 用户感知强的场景可以用“写后读主库”提升体验。
- 设计接口时要明确一致性承诺，避免调用方误用。
:::

## 15、CAP 理论是什么
CAP 理论用于描述分布式系统在一致性、可用性和分区容错性之间的取舍。网络分区发生时，系统通常无法同时保证强一致性和高可用性。

::: details 详情
### 三个概念

- C：Consistency，一致性，所有节点读取到的数据一致。
- A：Availability，可用性，每个请求都能得到非错误响应。
- P：Partition Tolerance，分区容错性，网络分区时系统仍能继续运行。

在分布式系统中，网络分区无法完全避免，因此 P 通常必须考虑。

### CP 系统

CP 更偏向一致性。

当网络分区发生时，为了避免读写到不一致数据，部分节点可能拒绝服务。

适合对正确性要求高的场景，例如配置中心、分布式锁、账务核心数据。

### AP 系统

AP 更偏向可用性。

当网络分区发生时，系统尽量继续响应请求，但不同节点之间可能短时间数据不一致。

适合允许最终一致的场景，例如动态、点赞、消息通知、缓存同步。

### 注意事项

- CAP 不是说平时只能三选二，而是网络分区发生时要做取舍。
- 一致性也有强一致、顺序一致、最终一致等不同层次。
- 具体系统常在不同业务链路上做不同取舍。
- 面试回答要结合业务场景，不要只背定义。
:::

## 16、BASE 理论是什么
BASE 理论是分布式系统中对强一致性的放松，强调基本可用、软状态和最终一致性。它常用于高可用、高并发但允许短暂不一致的业务场景。

::: details 详情
### 三个含义

BASE 包括：

- Basically Available：基本可用。
- Soft State：软状态。
- Eventually Consistent：最终一致。

它不是放弃一致性，而是接受系统在短时间内处于中间状态，最终通过同步、补偿和对账达到一致。

### 基本可用

基本可用是指故障或高峰时系统仍能提供核心能力。

例如：

- 非核心功能降级。
- 页面展示缓存数据。
- 限制部分用户访问。
- 排队处理请求。

### 软状态

软状态表示系统允许存在中间状态。

例如订单已创建但支付状态还在同步中，搜索索引还没更新到最新数据。

### 最终一致

最终一致表示经过一段时间后，多个副本或多个系统中的数据会收敛到一致。

常见手段：

- 消息队列。
- 定时任务。
- 重试补偿。
- 对账修复。

### 注意事项

- BASE 适合允许短暂不一致的业务，不适合所有场景。
- 要明确用户能看到哪些中间状态。
- 必须设计补偿、重试和对账机制。
- 最终一致要有时间边界和监控，不是无限期等待。
:::

## 17、系统降级和兜底方案如何设计
系统降级是在压力过大、依赖故障或部分能力不可用时，主动关闭或弱化非核心功能，保证核心链路继续可用。兜底方案用于在失败时提供可接受的替代结果。

::: details 详情
### 降级对象

常见降级对象包括：

- 推荐系统。
- 排行榜。
- 评论区。
- 非核心活动位。
- 第三方服务。
- 复杂查询和统计。

核心交易、支付、鉴权等链路通常不能随意降级，需要更严格的容灾方案。

### 兜底方式

常见兜底方式：

- 返回缓存数据。
- 返回默认配置。
- 使用静态文案。
- 隐藏非核心模块。
- 展示稍后重试。
- 切换备用服务。

例如推荐接口失败时，可以返回热门榜单或默认内容。

### 触发条件

降级可以由多种条件触发：

- 错误率升高。
- 响应时间超过阈值。
- 下游服务不可用。
- 线程池或连接池耗尽。
- 人工开关。
- 灰度规则。

### 注意事项

- 降级开关要支持快速生效和回滚。
- 降级结果要对用户可理解，不要静默失败。
- 要记录降级日志和指标，方便复盘。
- 降级不能掩盖长期容量问题，事后仍要修复根因。
:::

## 18、分库分表后如何做数据迁移
分库分表后的数据迁移通常涉及历史数据搬迁、增量数据同步、双写校验和流量切换。目标是在尽量不停机的情况下完成平滑迁移。

::: details 详情
### 常见步骤

一般可以分为：

- 设计新的分片规则。
- 准备目标库表。
- 全量迁移历史数据。
- 同步增量变更。
- 做数据校验。
- 灰度切读流量。
- 切写流量。
- 保留回滚窗口。

迁移过程要尽量可暂停、可重试、可回滚。

### 全量和增量

全量迁移用于搬迁已有数据。

增量同步用于处理迁移期间新产生或变更的数据。

增量同步常见方式：

- 业务双写。
- 读取 binlog。
- 消息队列同步。
- 定时任务补偿。

### 数据校验

迁移后要校验：

- 行数是否一致。
- 主键范围是否完整。
- 核心字段 hash 是否一致。
- 抽样业务查询是否正确。
- 增量延迟是否在可接受范围内。

### 注意事项

- 分片键一旦确定，后续调整成本很高。
- 双写期间要处理部分成功和重试幂等。
- 切流量应灰度进行，不要一次性全量切换。
- 迁移期间要监控延迟、错误率、数据库负载和数据差异。
:::

## 19、接口版本管理如何设计
接口版本管理用于在业务演进时兼容旧客户端和旧调用方，避免服务端变更直接破坏线上功能。它常见于 App、小程序、开放平台和多团队协作系统。

::: details 详情
### 为什么需要版本管理

接口变更可能包括：

- 字段新增。
- 字段删除。
- 字段含义变化。
- 参数格式变化。
- 错误码变化。
- 业务流程变化。

其中新增字段通常兼容性较好，删除字段和改变字段含义风险更高。

### 常见方式

常见版本设计：

- URL 路径版本：`/api/v1/users`。
- Header 版本：`Accept-Version: v1`。
- 查询参数版本：`?version=1`。
- 客户端版本配合灰度策略。

内部系统可以更轻量，开放平台通常要更严格。

### 兼容策略

接口升级时可以：

- 保持旧字段一段时间。
- 新增字段而不是改旧字段含义。
- 服务端根据版本返回不同结构。
- 提供迁移文档和废弃时间表。
- 对旧版本调用量做监控。

### 注意事项

- 不要频繁创建大版本，否则维护成本会急剧上升。
- 破坏性变更必须明确通知调用方。
- 移动端发版慢，要保留足够兼容窗口。
- 错误码、分页结构、时间格式也属于接口契约的一部分。
:::

## 20、服务发现是什么，解决什么问题
服务发现用于让服务调用方动态找到目标服务实例地址，避免把 IP 和端口写死在配置里。它常用于微服务、容器化和弹性扩缩容场景。

::: details 详情
### 为什么需要服务发现

在分布式系统中，服务实例可能会：

- 扩容。
- 缩容。
- 重启。
- 故障下线。
- 被调度到不同机器。

如果调用方写死实例地址，维护成本高，也无法及时避开故障实例。

### 基本流程

常见流程：

1. 服务启动后向注册中心注册自己的地址和元信息。
2. 服务定期上报心跳。
3. 调用方从注册中心获取可用实例列表。
4. 调用方结合负载均衡策略选择实例。
5. 异常实例被摘除。

### 常见实现

常见服务发现方案：

- Consul。
- Eureka。
- Nacos。
- ZooKeeper。
- Kubernetes Service 和 DNS。
- 云厂商服务发现。

### 客户端发现和服务端发现

客户端发现是调用方自己拉取实例列表并负载均衡。

服务端发现是请求先到网关或负载均衡器，再由它转发到目标实例。

### 注意事项

- 注册中心本身要高可用。
- 心跳和摘除策略要避免过慢或过于敏感。
- 调用方要有本地缓存，注册中心短暂不可用时仍能工作。
- 服务发现要和健康检查、负载均衡、熔断一起设计。
:::

## 21、配置中心如何设计
配置中心用于集中管理应用配置，让配置可以按环境、服务、租户或灰度规则动态下发，减少重新发布代码的成本。

::: details 详情
### 适合管理什么

配置中心常管理：

- 功能开关。
- 限流阈值。
- 降级开关。
- 灰度规则。
- 业务参数。
- 第三方服务地址。

敏感配置如密钥、证书也可以集中管理，但需要更严格的权限和加密策略。

### 核心能力

配置中心通常需要：

- 配置读取。
- 动态推送。
- 版本管理。
- 权限控制。
- 灰度发布。
- 审计日志。
- 回滚能力。

配置变更本身就是一次线上变更，应有记录和控制。

### 客户端设计

客户端需要考虑：

- 启动时拉取配置。
- 本地缓存配置。
- 监听配置变更。
- 配置中心不可用时使用最后一次可用配置。
- 配置解析失败时拒绝生效。

### 注意事项

- 配置变更要支持审批和回滚。
- 高风险配置要灰度发布，避免一次性影响全量用户。
- 配置项要有默认值和说明。
- 不要把频繁变化的大数据当配置中心使用。
:::

## 22、异步任务系统如何设计
异步任务系统用于把耗时、可延后或需要重试的工作从同步请求链路中拆出来，提升接口响应速度和系统稳定性。

::: details 详情
### 适合场景

常见异步任务包括：

- 发送短信或邮件。
- 生成报表。
- 大文件处理。
- 订单超时关闭。
- 数据同步。
- 图片转码。
- AI 任务执行。

这些任务通常不适合让用户在接口中一直等待。

### 基本结构

常见结构：

```txt
业务服务 -> 任务队列 -> Worker -> 结果存储
```

业务服务创建任务后快速返回任务 ID，前端或调用方通过查询接口获取任务状态。

### 任务状态

常见状态包括：

- pending：等待执行。
- running：执行中。
- success：执行成功。
- failed：执行失败。
- canceled：已取消。
- retrying：等待重试。

状态要可查询，也要方便排查问题。

### 可靠性设计

需要考虑：

- 任务幂等。
- 失败重试。
- 超时控制。
- 死信队列。
- 任务去重。
- 最大并发数。
- 任务优先级。

### 注意事项

- 不要无限重试，重试要有次数和退避策略。
- 任务参数要保存快照，避免后续数据变化导致结果不确定。
- Worker 要支持优雅关闭，避免任务执行一半丢失。
- 高风险任务要有审计日志和人工补偿入口。
:::

## 23、搜索系统如何设计
搜索系统用于支持关键词检索、筛选、排序和高亮等能力。常见架构是业务数据库负责写入和权威数据，搜索引擎负责检索和排序。

::: details 详情
### 基本架构

常见结构：

```txt
业务写库 -> 数据同步 -> 搜索索引 -> 查询服务
```

业务数据库保存权威数据，Elasticsearch、OpenSearch 等搜索引擎保存适合检索的索引。

### 数据同步

常见同步方式：

- 业务双写。
- 消息队列。
- 监听 binlog。
- 定时全量同步。
- 增量任务补偿。

搜索索引通常允许最终一致，但要控制延迟和失败补偿。

### 查询能力

搜索系统通常需要：

- 分词。
- 过滤条件。
- 排序规则。
- 分页。
- 高亮。
- 权重调整。
- 拼写纠错或同义词。

排序可能综合相关性、时间、热度、销量、用户偏好等因素。

### 注意事项

- 搜索结果和数据库可能短暂不一致，要有业务预期。
- 深分页成本高，可以使用 search_after 或游标方案。
- 索引结构变更要考虑重建和双索引切换。
- 搜索质量需要通过点击率、转化率和人工评估持续优化。
:::

## 24、历史数据归档如何设计
历史数据归档用于把低频访问的旧数据从在线库中迁移到成本更低、查询压力更小的存储中，从而控制在线表体积并提升核心查询性能。

::: details 详情
### 为什么需要归档

业务运行时间变长后，在线表可能出现：

- 数据量持续膨胀。
- 索引变大。
- 查询变慢。
- 备份恢复时间变长。
- 存储成本上升。

如果大量历史数据很少访问，就可以考虑归档。

### 归档策略

常见策略：

- 按时间归档，例如保留近 12 个月在线数据。
- 按状态归档，例如已完成、已关闭订单。
- 按租户或业务线归档。
- 冷热分层存储。

归档规则要和业务、合规、审计要求对齐。

### 归档流程

典型流程：

1. 筛选待归档数据。
2. 分批迁移到归档库或对象存储。
3. 校验迁移结果。
4. 删除或标记在线库数据。
5. 提供历史查询入口。

整个过程要限速，避免影响在线业务。

### 注意事项

- 归档前要确认数据是否仍会被在线流程依赖。
- 删除在线数据前必须校验归档完整性。
- 历史查询要说明延迟和可用范围。
- 归档任务要可恢复、可重试、可审计。
:::

## 25、系统中的幂等键应该如何设计
幂等键用于标识一次唯一操作，避免网络重试、前端重复提交或消息重复投递导致同一动作被执行多次。

::: details 详情
### 为什么需要幂等键

在分布式系统中，同一个请求可能因为：

- 用户重复点击。
- 网络超时重试。
- 网关重发。
- 消息队列重复投递。
- 客户端恢复后重复提交。

被执行多次。

幂等键就是用来识别“同一次业务操作”的。

### 常见设计

常见做法：

- 客户端生成 requestId。
- 服务端生成业务单号。
- 使用业务唯一字段组合。
- 使用 Redis 或数据库记录已处理请求。

幂等键通常要和业务场景绑定，而不是全局随便复用。

### 存储和校验

可以把幂等键存到：

- 数据库唯一索引。
- Redis。
- 去重表。
- 状态机表。

处理流程一般是：

1. 先检查幂等键是否已存在。
2. 已存在则直接返回已有结果。
3. 不存在则执行业务逻辑。
4. 处理成功后写入幂等记录。

### 注意事项

- 幂等键要设置作用范围和过期时间。
- 要考虑并发竞争，不能只靠先查后写。
- 重要业务最好有数据库唯一约束兜底。
- 幂等键命中后返回的结果应尽量稳定可复用。
:::

## 26、分布式限流系统如何设计
分布式限流用于在多实例部署下统一控制接口、用户、租户或 IP 的请求速率，防止流量激增拖垮系统。

::: details 详情
### 为什么需要分布式限流

单机限流只能控制单个实例的流量。

在多实例环境中，每个实例都单独限流可能导致总流量仍然过高。

分布式限流可以从全局视角控制总体请求量。

### 常见算法

常见算法包括：

- 令牌桶。
- 漏桶。
- 固定窗口计数。
- 滑动窗口计数。
- 滑动日志。

令牌桶适合允许一定突发的场景。

### 实现方式

常见实现依赖：

- Redis 原子计数。
- Lua 脚本。
- 网关限流。
- 本地缓存 + 全局校准。
- 消息队列削峰。

例如可以按用户、接口、租户和 IP 维度分别限流。

### 注意事项

- 限流键设计要合理，避免粒度过粗或过细。
- 要处理 Redis 故障时的降级策略。
- 限流返回要友好，最好带重试建议。
- 核心链路和非核心链路限流阈值应不同。
:::

## 27、缓存预热系统如何设计
缓存预热是指在流量高峰前提前把热点数据加载进缓存，减少首次访问击穿数据库的风险，提升用户体验和系统稳定性。

::: details 详情
### 为什么需要预热

如果热门页面、活动页或核心配置第一次被大量用户访问时才加载缓存，就可能出现：

- 数据库压力骤增。
- 首次访问延迟高。
- 热点 key 同时回源。
- 缓存击穿或雪崩。

### 预热对象

常见预热对象：

- 热点商品详情。
- 活动配置。
- 首页推荐数据。
- 配置字典。
- 常用字典表。
- 热门榜单。

### 预热方式

常见方式：

- 启动时加载。
- 发布后定时任务预热。
- 流量前手动预热。
- 基于访问日志预热。
- 按活动时间自动预热。

预热过程要限速，避免一次性压垮后端。

### 注意事项

- 缓存预热的数据要和业务版本匹配。
- 预热失败要有告警和重试。
- 不要把所有数据都预热，优先选高频和高价值数据。
- 预热后仍要设置合理过期和回源策略。
:::

## 28、灰度发布系统如何设计
灰度发布用于把新版本逐步暴露给部分用户，降低全量发布风险，并通过监控数据判断是否继续放量。

::: details 详情
### 核心目标

灰度发布希望实现：

- 小流量验证。
- 快速发现问题。
- 快速回滚。
- 用户分群控制。

### 常见灰度维度

常见灰度维度：

- 用户 ID。
- 地区。
- 设备类型。
- 租户。
- 版本号。
- 流量比例。

例如可以让 5% 用户先进入新版本。

### 发布链路

常见流程：

1. 发布新版本但不全量放开。
2. 只让灰度用户命中新版本。
3. 观察错误率、延迟、转化率和核心业务指标。
4. 指标正常后逐步扩大比例。
5. 有问题则快速回滚。

### 注意事项

- 灰度规则要稳定，否则同一用户会频繁切换版本。
- 需要同时控制前端资源、后端接口和配置版本。
- 灰度期间要重点关注监控和告警。
- 回滚要能快速切回旧版本，不只是代码回退。
:::

## 29、消息队列如何用于削峰
消息队列削峰是指在高并发请求到来时，先把任务写入队列，再由消费者按系统承载能力慢慢处理，避免瞬时流量直接打垮后端。

::: details 详情
### 适合场景

消息队列削峰适合：

- 秒杀下单。
- 批量通知。
- 数据同步。
- 日志处理。
- 异步报表。
- AI 任务排队。

这些场景通常允许短时间排队，而不要求所有请求同步完成。

### 基本结构

常见结构：

```txt
请求入口 -> 校验和限流 -> 写入队列 -> 消费者处理 -> 状态查询
```

前端或调用方拿到任务 ID 后，可以查询处理结果。

### 关键设计

需要考虑：

- 队列长度。
- 消费者并发数。
- 消息重试。
- 死信队列。
- 消息幂等。
- 任务超时。
- 处理结果通知。

### 注意事项

- 队列不是无限容量，仍需要入口限流。
- 消费速度不足时会产生积压，要有告警。
- 用户体验上要明确“排队中”或“处理中”状态。
- 关键业务要保证消息不丢、处理幂等、失败可补偿。
:::

## 30、分布式 ID 生成器如何设计
分布式 ID 生成器用于在多实例、多服务环境下生成全局唯一 ID。常见目标是唯一、趋势递增、高性能、可用性高，并且方便排查。

::: details 详情
### 常见方案

常见实现包括：

- 数据库自增 ID。
- UUID。
- Redis 自增。
- Snowflake 类算法。
- 号段模式。

数据库自增简单但扩展性有限，UUID 分散且不利于索引，Snowflake 和号段模式更适合高并发业务。

### Snowflake 思路

典型 Snowflake ID 会包含：

- 时间戳。
- 机器 ID。
- 序列号。

这样可以在不同机器上并发生成不冲突的 ID，并保持大致按时间递增。

### 注意事项

- 机器 ID 不能冲突。
- 系统时钟回拨要有处理策略。
- ID 长度要和数据库字段类型匹配。
- 如果需要暴露给用户，要避免泄露业务规模。
- 强依赖趋势递增时，要关注热点写入问题。
:::

## 31、配置中心如何设计
配置中心用于集中管理应用配置，让配置可以在不重新发版的情况下变更和下发。它适合管理开关、阈值、灰度规则和环境配置。

::: details 详情
### 核心能力

配置中心通常需要：

- 配置存储。
- 权限控制。
- 版本管理。
- 变更审批。
- 实时推送。
- 回滚能力。
- 客户端缓存。

配置变更要可追踪、可审计、可恢复。

### 下发方式

常见下发方式包括：

- 客户端定时拉取。
- 长轮询。
- WebSocket 或长连接推送。
- 服务启动时加载。

对实时性要求高的配置可以使用推送，普通配置可以定时拉取。

### 容错设计

客户端要考虑配置中心不可用的情况：

- 使用本地缓存。
- 使用默认值。
- 保留上一次成功配置。
- 配置解析失败时拒绝覆盖。

不能因为配置中心短暂故障导致业务整体不可用。

### 注意事项

- 敏感配置要加密存储和传输。
- 配置变更要有灰度发布能力。
- 配置项要有命名规范和负责人。
- 高频变更配置要接入监控，避免误操作放大事故。
:::

## 32、审计日志系统如何设计
审计日志用于记录关键操作的主体、对象、时间、来源和结果，帮助追踪责任、满足合规要求，并在安全事件后还原现场。

::: details 详情
### 需要记录什么

常见字段包括：

- 操作人。
- 操作时间。
- 操作类型。
- 操作对象。
- 请求来源 IP。
- 设备信息。
- 操作前后的关键值。
- 成功或失败结果。
- requestId 或 traceId。

审计日志要能回答“谁在什么时候对什么做了什么”。

### 写入方式

可以同步写入，也可以异步写入。

高并发系统通常会先写消息队列，再由日志服务消费落库。

这样可以降低主业务链路耗时。

### 存储和查询

审计日志通常要求：

- 不可随意修改。
- 保留足够长时间。
- 支持按用户、对象、时间范围查询。
- 支持导出和告警。

重要日志可以写入专门的日志系统或冷存储。

### 注意事项

- 敏感字段要脱敏或加密。
- 审计日志本身也要有访问权限控制。
- 不能只依赖前端上报，关键操作要在服务端记录。
- 日志格式要稳定，方便后续检索和合规审查。
:::

## 33、大文件上传系统如何设计
大文件上传系统要解决上传时间长、网络中断、服务端压力和文件完整性校验问题。常见方案是分片上传、断点续传和合并校验。

::: details 详情
### 基本流程

常见流程：

1. 客户端计算文件 hash。
2. 向服务端申请上传任务。
3. 客户端按固定大小切分文件。
4. 并发上传分片。
5. 服务端记录已上传分片。
6. 所有分片完成后合并文件。
7. 校验最终文件完整性。

如果任务已存在，可以直接跳过已上传分片。

### 关键能力

系统需要支持：

- 分片上传。
- 断点续传。
- 秒传。
- 并发控制。
- 上传进度。
- 失败重试。
- 文件类型和大小校验。

服务端要限制单用户并发和总存储占用。

### 存储方式

文件可以先上传到业务服务器，也可以直传对象存储。

大规模场景通常使用对象存储直传，业务服务只负责签名、任务管理和元数据记录。

这样可以减少应用服务器带宽压力。

### 注意事项

- 文件 hash 计算会消耗客户端 CPU。
- 分片大小要在并发和请求数量之间平衡。
- 合并失败要能重试和清理临时分片。
- 上传接口要做鉴权、防刷和病毒扫描。
:::
