# 系统设计

这里用于整理系统设计相关面试题，后续按限流、熔断、降级、缓存一致性、分布式锁、消息队列、幂等、链路追踪、高并发读写等方向追加。

## 1、系统为什么需要限流，常见限流算法有哪些
限流用于限制单位时间内的请求数量，防止流量突增打垮服务。它常用于接口保护、登录防刷、短信发送、支付下单、网关入口等场景。

::: details 详情
### 为什么需要限流

如果没有限流，突发流量可能导致：

- CPU 或内存打满。
- 数据库连接池耗尽。
- 下游服务被压垮。
- 队列积压。
- 正常用户请求被拖慢。

限流的目标不是让所有请求都成功，而是在系统承载范围内保护核心链路。

### 固定窗口算法

把时间按固定窗口切分，例如每分钟最多 100 次。

优点：实现简单。

缺点：窗口边界可能出现流量突刺。

### 滑动窗口算法

记录更细粒度的时间片或请求时间，按最近一段时间统计请求数。

优点：比固定窗口更平滑。

缺点：实现和存储成本更高。

### 漏桶算法

请求进入桶中，按照固定速率流出。

优点：输出流量稳定。

缺点：突发流量处理能力较弱。

### 令牌桶算法

系统按固定速率生成令牌，请求必须拿到令牌才能通过。

优点：既能限制平均速率，也允许一定突发流量。

### 面试要点

- 限流可以放在网关、服务端、用户维度、IP 维度或业务维度。
- 高并发分布式场景常结合 Redis 或网关限流。
- 限流后要返回明确错误码，并配合降级、排队或重试提示。
- 限流阈值需要结合容量压测和业务优先级确定。
:::

## 2、熔断和降级有什么区别
熔断和降级都是系统稳定性手段。熔断更关注在下游异常时快速切断调用，避免故障扩散；降级更关注在资源不足或功能异常时牺牲非核心能力，保证核心链路可用。

::: details 详情
### 熔断

熔断类似电路保险丝。

当某个下游服务错误率或超时率过高时，上游不再继续请求它，而是直接返回失败或兜底结果。

常见状态：

- closed：正常调用。
- open：熔断打开，直接拒绝或兜底。
- half-open：放少量请求探测下游是否恢复。

### 降级

降级是主动降低服务能力，例如：

- 首页不展示推荐模块。
- 搜索只返回缓存结果。
- 评论区暂时关闭。
- 非核心接口返回默认值。
- 暂停低优先级任务。

降级可以是自动触发，也可以由运营或研发手动开关触发。

### 二者区别

- 熔断通常针对下游依赖异常。
- 降级通常针对整体资源压力或非核心能力取舍。
- 熔断强调保护调用链。
- 降级强调保核心、舍非核心。

### 组合使用

实际系统中二者经常一起使用：

```txt
下游推荐服务超时
-> 熔断推荐服务调用
-> 降级为默认推荐列表
```

### 面试要点

- 熔断需要指标：错误率、超时率、并发数等。
- 降级需要明确核心链路和非核心链路。
- 兜底结果要可接受，不能产生严重业务误导。
- 熔断和降级都需要监控、告警和恢复策略。
:::

## 3、接口幂等性如何设计
接口幂等性是指同一个请求执行一次和执行多次，最终业务结果一致。它常用于下单、支付回调、消息消费、表单提交等场景。

::: details 详情
### 为什么需要幂等

请求可能因为很多原因重复到达：

- 用户重复点击。
- 网络超时后重试。
- 网关重试。
- 消息队列重复投递。
- 第三方回调重复通知。

如果没有幂等控制，可能出现重复下单、重复扣款、重复发货等问题。

### 幂等键

常见做法是为请求生成业务幂等键：

```txt
userId + orderNo
```

或：

```txt
requestId
```

服务端收到请求后，先检查幂等记录，已经处理过就直接返回之前结果。

### 常见实现方式

- 数据库唯一索引。
- 业务状态机。
- Redis `SET NX`。
- 幂等请求表。
- 分布式锁。
- 消息消费记录表。

### 数据库唯一索引示例

```sql
create unique index uk_order_no on orders(order_no);
```

如果同一个订单号重复插入，数据库会拒绝重复数据。

### 注意事项

- 幂等键要和业务语义绑定。
- 不能只靠前端禁用按钮。
- 重试时要能返回相同业务结果。
- 支付、库存等关键场景要结合事务、锁和状态机。
:::

## 4、缓存和数据库如何保证一致性
缓存和数据库一致性问题通常出现在写入数据库后，缓存中的旧数据仍被读取。实际系统中很难做到绝对强一致，更多是根据业务选择最终一致方案。

::: details 详情
### 常见策略

常见做法包括：

- 先更新数据库，再删除缓存。
- 先删除缓存，再更新数据库。
- 延迟双删。
- 通过消息队列异步删除缓存。
- 订阅数据库变更同步缓存。

### 推荐常见做法

多数业务更常用：

```txt
先更新数据库 -> 再删除缓存
```

下次读取时缓存不存在，再从数据库加载新数据并写回缓存。

### 为什么不是更新缓存

直接更新缓存可能遇到：

- 并发写入导致覆盖。
- 缓存结构和数据库结构不一致。
- 部分缓存组合数据难以准确更新。

删除缓存通常比更新缓存更简单、更稳。

### 延迟双删

延迟双删流程：

```txt
删除缓存 -> 更新数据库 -> 延迟一小段时间 -> 再删除缓存
```

它用于降低并发读写导致旧值回写缓存的概率。

### 注意事项

- 缓存一致性要结合业务容忍度设计。
- 对强一致要求高的业务，不应只依赖缓存方案。
- 删除缓存失败要有重试或补偿机制。
- 缓存要设置过期时间，避免长期脏数据。
:::

## 5、消息队列如何保证消息可靠性
消息队列可靠性通常要从生产者、队列本身和消费者三个环节考虑，目标是尽量避免消息丢失，并能处理重复消息和消费失败。

::: details 详情
### 生产者可靠性

生产者发送消息后，需要确认消息是否成功进入队列。

常见手段：

- 发送确认机制。
- 失败重试。
- 本地消息表。
- 事务消息。

如果业务数据已经写入数据库，但消息发送失败，就需要补偿机制。

### 队列可靠性

队列自身要保证消息不会因为服务异常轻易丢失。

常见手段：

- 消息持久化。
- 主从复制。
- 高可用集群。
- 合理的 ACK 机制。

不同消息队列实现细节不同，要结合具体产品理解。

### 消费者可靠性

消费者处理消息后再确认消费成功。

如果处理失败，不应过早 ACK，否则消息可能丢失。

常见处理：

- 消费失败重试。
- 重试次数限制。
- 死信队列。
- 消费幂等。

### 重复消息

即使可靠性设计充分，也要接受消息可能重复。

所以消费者必须做幂等，例如用：

```txt
messageId
businessId + eventType
```

作为消费记录唯一键。

### 注意事项

- 可靠性越高，系统复杂度和成本通常越高。
- 重试要避免无限重试拖垮系统。
- 死信队列要有人处理和告警。
- 消息顺序、延迟和可靠性之间可能存在取舍。
:::

## 6、分布式链路追踪有什么作用
分布式链路追踪用于跟踪一次请求在多个服务之间的调用路径，帮助排查慢接口、错误来源和服务依赖关系。

::: details 详情
### 为什么需要链路追踪

在单体应用中，一个请求通常只经过一个服务。

在微服务系统中，一个请求可能经过：

```txt
网关 -> 用户服务 -> 订单服务 -> 库存服务 -> 支付服务
```

如果请求变慢或失败，需要知道问题出在哪个服务、哪个接口、哪次调用。

### 核心概念

常见概念包括：

- Trace：一次完整请求链路。
- Span：链路中的一次具体操作或调用。
- TraceId：标识一次完整请求。
- SpanId：标识某个调用片段。
- ParentSpanId：标识父调用关系。

### 日志关联

服务日志中通常会打印 `traceId`：

```txt
traceId=abc123 userId=1 path=/api/orders cost=120ms
```

这样可以把分散在多个服务中的日志串起来。

### 常见用途

- 定位慢请求。
- 分析调用依赖。
- 排查错误传播路径。
- 统计接口耗时。
- 辅助容量评估。

### 注意事项

- TraceId 要在服务间透传。
- 异步任务、消息队列也要传递链路上下文。
- 采样率要平衡成本和可观测性。
- 链路追踪要和日志、指标、告警结合使用。
:::

## 7、高并发读写系统如何设计
高并发读写系统设计要先区分读多写少、写多读少还是读写都高，再围绕缓存、数据库、队列、分片、限流和降级做组合设计。

::: details 详情
### 先分析业务模型

设计前要明确：

- QPS 大概多少。
- 读写比例。
- 数据一致性要求。
- 是否有热点数据。
- 是否允许异步处理。
- 峰值流量来源。

不同业务模型对应不同方案。

### 读多写少

读多写少常见方案：

- CDN。
- 本地缓存。
- Redis 缓存。
- 读写分离。
- 页面静态化。
- 热点数据预热。

重点是降低数据库读压力。

### 写多场景

写多场景常见方案：

- 消息队列削峰。
- 批量写入。
- 分库分表。
- 异步处理。
- 写入限流。
- 幂等控制。

重点是保护数据库和下游服务。

### 热点问题

热点 key、热点商品、热点接口可能导致局部瓶颈。

常见处理：

- 热点缓存。
- 多级缓存。
- 请求合并。
- 分片 key。
- 限流和降级。

### 注意事项

- 高并发不是只加缓存。
- 一致性要求越高，设计越复杂。
- 队列削峰会牺牲实时性。
- 所有方案都要配合监控、压测和容量评估。
:::

## 8、分库分表适合解决什么问题
分库分表用于解决单库或单表在数据量、写入吞吐、存储容量和查询性能上的瓶颈。它不是默认方案，通常是在垂直优化和索引优化后仍无法满足需求时考虑。

::: details 详情
### 分库

分库是把数据拆到多个数据库实例中。

常见目的：

- 分摊存储压力。
- 分摊连接和写入压力。
- 降低单实例故障影响。

### 分表

分表是把一张大表拆成多张小表。

例如订单表：

```txt
orders_00
orders_01
orders_02
...
```

可以按照用户 ID、订单 ID、时间等规则拆分。

### 分片键

分片键选择很关键。

好的分片键应该：

- 查询中经常出现。
- 分布相对均匀。
- 不容易频繁变化。
- 能避免热点集中。

### 带来的问题

分库分表会引入很多复杂度：

- 跨库 join 困难。
- 分布式事务复杂。
- 全局唯一 ID。
- 分页和排序复杂。
- 扩容和迁移成本高。
- 数据聚合查询困难。

### 注意事项

- 分库分表是复杂度很高的方案，不要过早使用。
- 能用索引、缓存、读写分离解决的，不一定要分库分表。
- 分片规则要提前考虑扩容。
- 后台管理、报表和数据分析通常需要额外方案。
:::

## 9、分布式系统如何生成全局唯一 ID
全局唯一 ID 用于在分布式系统中为订单、用户、消息等数据生成唯一标识。常见方案包括 UUID、数据库号段、Redis 自增和雪花算法。

::: details 详情
### UUID

UUID 生成简单，不依赖中心服务。

优点：

- 本地生成。
- 冲突概率极低。

缺点：

- 字符串较长。
- 不递增。
- 作为数据库主键可能影响索引局部性。

### 数据库自增

单库自增 ID 简单可靠，但分布式场景会有单点和性能瓶颈。

可以使用号段模式：

```txt
服务一次申请一段 ID，例如 1000 到 1999
```

服务在本地发号，号段用完后再申请。

### Redis 自增

使用 Redis `INCR` 可以生成递增 ID。

优点是简单，缺点是依赖 Redis 可用性，并需要考虑持久化和高可用。

### 雪花算法

雪花算法通常由：

- 时间戳。
- 机器 ID。
- 序列号。

组合生成 64 位整数。

它可以本地生成，趋势递增，适合高并发场景。

### 注意事项

- ID 是否需要递增会影响方案选择。
- 机器 ID 冲突会导致 ID 冲突。
- 雪花算法要处理时钟回拨。
- 数据库主键要考虑索引性能和存储成本。
:::

## 10、负载均衡常见策略有哪些
负载均衡用于把请求分发到多台后端实例，提升系统吞吐、可用性和扩展能力。常见策略包括轮询、加权轮询、最少连接、哈希和一致性哈希。

::: details 详情
### 常见策略

- 轮询：按顺序把请求分发给后端实例。
- 加权轮询：给性能更强的实例更高权重。
- 最少连接：优先分发给当前连接数更少的实例。
- IP Hash：同一客户端 IP 尽量落到同一实例。
- 一致性哈希：节点增删时尽量减少请求迁移范围。

### 如何选择

普通无状态服务可以使用轮询或加权轮询。

长连接服务更适合关注连接数、连接时长和实例负载。

需要会话粘滞时，可以使用 IP Hash 或基于 Cookie 的路由，但更推荐把会话状态外置到 Redis、数据库或独立 Session 服务中。

### 健康检查

负载均衡不只是分发请求，还要识别不可用实例。

健康检查通常包括：

- TCP 端口是否可连接。
- HTTP 健康检查接口是否返回成功。
- 实例是否处于启动、下线或熔断状态。
- 错误率和响应时间是否异常。

### 注意事项

- 后端服务应尽量无状态，降低路由粘滞依赖。
- 权重调整要结合监控数据，不要凭感觉配置。
- 健康检查过于频繁会增加压力，过于宽松会放大故障。
- 负载均衡层本身也要避免单点故障。
:::

## 11、数据库读写分离如何设计
读写分离是指写请求进入主库，读请求尽量走从库，用来降低主库压力并提升读请求吞吐。它适合读多写少、读流量明显大于写流量的系统。

::: details 详情
### 基本结构

常见结构是：

```txt
业务服务 -> 数据访问层 -> 主库写入
                  -> 从库读取
```

主库负责写入和复制日志，从库通过复制机制同步主库数据。

### 路由规则

读写分离通常在以下位置实现：

- 业务代码中区分读写数据源。
- ORM 或数据库中间件自动路由。
- 数据库代理层统一转发。

写操作、事务内查询和强一致读通常走主库。

普通列表、详情页、统计类读请求可以走从库。

### 主从延迟

读写分离最大的问题是主从复制延迟。

例如用户刚修改资料，马上读取详情，如果读到从库，可能看到旧数据。

常见解决方式：

- 写后短时间内读主库。
- 关键业务强制读主库。
- 根据复制延迟动态摘除落后从库。
- 前端或服务端做短暂状态补偿。

### 注意事项

- 读写分离不能提升写入能力，写瓶颈仍在主库。
- 从库数量越多，复制和运维复杂度越高。
- 事务一致性要求高的场景要谨慎读从库。
- 需要监控复制延迟、主从状态和从库错误率。
:::

## 12、系统中的热点数据如何治理
热点数据是指短时间内被大量访问或更新的数据，例如热门商品、爆款活动、明星动态和秒杀库存。治理热点的核心是识别热点、分散压力、保护后端。

::: details 详情
### 热点类型

常见热点包括：

- 热点读：大量请求读取同一份数据。
- 热点写：大量请求更新同一个 key 或同一行记录。
- 热点缓存 key：大量流量集中到一个缓存 key。
- 热点分片：分库分表后某个分片压力明显更高。

### 热点读治理

热点读可以通过缓存和多级缓存缓解：

- CDN 缓存静态内容。
- 本地缓存减少远程缓存访问。
- Redis 缓存热点数据。
- 热点 key 副本拆分。
- 提前预热缓存。

对读多写少的数据，可以适当延长缓存时间。

### 热点写治理

热点写更难处理，因为写请求往往需要一致性。

常见方式：

- 请求排队削峰。
- 分桶计数后异步汇总。
- 使用消息队列串行化处理。
- 将强一致写改为最终一致。
- 对超出容量的请求做限流或降级。

### 识别热点

热点治理依赖监控：

- 接口 QPS。
- 缓存 key 访问频率。
- 数据库行锁等待。
- 分片负载。
- 错误率和响应时间。

### 注意事项

- 不要只靠扩容解决热点写，单点更新仍可能成为瓶颈。
- 热点 key 失效要防止缓存击穿。
- 秒杀类场景要把限流、库存预扣和队列一起设计。
- 热点规则要动态化，避免只依赖人工配置。
