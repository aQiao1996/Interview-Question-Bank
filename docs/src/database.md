# 数据库

这里用于整理数据库相关面试题，后续按 SQL、索引、事务、锁、MVCC、慢查询优化、分页优化、连接池、Redis 等方向追加。

## 1、数据库索引为什么能提升查询速度
数据库索引是一种辅助数据结构，可以帮助数据库更快定位数据，避免对整张表做全表扫描。常见关系型数据库中，B+ 树索引非常常见。

::: details 详情
### 没有索引时

如果没有索引，查询可能需要逐行扫描：

```sql
select * from users where name = 'Tom';
```

数据量越大，全表扫描成本越高。

### 有索引时

如果 `name` 字段有索引，数据库可以先在索引结构中查找 `Tom` 对应的位置，再回表读取完整数据。

这类似查字典时先看目录，而不是从第一页翻到最后一页。

### B+ 树索引特点

B+ 树适合数据库索引，原因包括：

- 树高较低，减少磁盘 IO。
- 叶子节点有序，适合范围查询。
- 非叶子节点只保存索引键，能容纳更多索引项。
- 查询、插入、删除都有较稳定的复杂度。

### 索引的代价

索引不是越多越好：

- 会占用额外存储空间。
- 插入、更新、删除时需要维护索引。
- 低选择性字段建索引收益可能不高。
- 复杂查询中索引可能不被优化器选择。

### 常见适合建索引的字段

- 经常出现在 `where` 条件中的字段。
- 经常用于 `join` 的字段。
- 经常用于 `order by` 或 `group by` 的字段。
- 高区分度字段，例如手机号、订单号。

### 面试要点

- 索引通过减少扫描范围提升查询性能。
- 索引会降低写入性能，需要权衡。
- 联合索引要注意最左前缀原则。
- 是否使用索引要结合执行计划分析。
:::

## 2、数据库事务的 ACID 分别是什么
事务是一组要么全部成功、要么全部失败的数据库操作。ACID 是事务的四个核心特性：原子性、一致性、隔离性和持久性。

::: details 详情
### Atomicity 原子性

原子性表示事务中的操作要么全部执行成功，要么全部回滚。

例如转账：

```txt
A 扣款 100
B 加款 100
```

这两个操作必须一起成功或一起失败。

### Consistency 一致性

一致性表示事务执行前后，数据库都要满足业务约束和数据约束。

例如账户余额不能无故增加或减少，唯一索引不能被破坏。

### Isolation 隔离性

隔离性表示多个事务并发执行时，彼此之间不能产生不符合隔离级别预期的干扰。

常见并发问题包括：

- 脏读。
- 不可重复读。
- 幻读。

不同隔离级别解决的问题不同。

### Durability 持久性

持久性表示事务提交后，数据变更应该被持久保存，即使数据库宕机，也应该能通过日志等机制恢复。

### 事务示例

```sql
begin;

update accounts set balance = balance - 100 where id = 1;
update accounts set balance = balance + 100 where id = 2;

commit;
```

如果中间失败，需要执行 `rollback`。

### 面试要点

- 原子性关注全部成功或全部失败。
- 一致性关注业务约束和数据约束不被破坏。
- 隔离性关注并发事务之间的影响。
- 持久性关注提交后的数据可靠保存。
:::

## 3、MVCC 是什么，解决了什么问题
MVCC 是多版本并发控制，通过为数据维护多个版本，让读写操作在很多场景下可以并发执行，减少锁竞争，提高数据库并发能力。

::: details 详情
### 为什么需要 MVCC

如果读写都依赖互斥锁，并发性能会很差：

```txt
读阻塞写
写阻塞读
```

MVCC 的核心目标是让读操作尽量不阻塞写操作，写操作也尽量不阻塞普通一致性读。

### 基本思想

数据更新时，不是简单覆盖旧值，而是形成新版本。

事务读取数据时，根据自己的可见性规则读取某个版本。

在 InnoDB 中，MVCC 通常和：

- undo log。
- Read View。
- 事务 ID。
- 隐藏字段。

这些机制有关。

### 快照读和当前读

快照读读取的是事务可见的数据版本，例如普通 `select`。

当前读读取的是最新数据，并可能加锁，例如：

```sql
select * from users where id = 1 for update;
```

### 解决的问题

MVCC 可以减少：

- 读写阻塞。
- 锁竞争。
- 并发事务之间的不必要等待。

在不同隔离级别下，MVCC 的可见性规则会不同。

### 注意事项

- MVCC 不代表完全不需要锁。
- 更新、删除、当前读仍可能加锁。
- 长事务会导致旧版本长期无法清理。
- MVCC 和隔离级别、undo log、事务可见性要结合理解。
:::

## 4、数据库事务隔离级别有哪些
事务隔离级别用于控制并发事务之间的可见性和影响范围。隔离级别越高，并发异常越少，但性能和并发能力可能越低。

::: details 详情
### 常见并发问题

- 脏读：读到其他事务未提交的数据。
- 不可重复读：同一事务内两次读取同一行，结果不同。
- 幻读：同一事务内两次范围查询，结果行数不同。

### Read Uncommitted

读未提交，可能读到其他事务未提交的数据。

它隔离性最低，实际业务中很少使用。

### Read Committed

读已提交，只能读到其他事务已经提交的数据。

可以避免脏读，但可能出现不可重复读。

### Repeatable Read

可重复读，保证同一事务内多次读取同一数据结果一致。

MySQL InnoDB 默认隔离级别通常是 Repeatable Read。

### Serializable

串行化，隔离级别最高，事务像串行执行一样。

它可以避免大多数并发问题，但并发性能最低。

### 对比

```txt
Read Uncommitted -> Read Committed -> Repeatable Read -> Serializable
隔离性逐渐增强，并发能力通常逐渐降低
```

### 面试要点

- 隔离级别是性能和一致性的权衡。
- 不同数据库对隔离级别的实现细节可能不同。
- MySQL InnoDB 的 Repeatable Read 通常结合 MVCC 和锁机制。
- 业务中要根据一致性要求选择隔离级别，而不是盲目使用最高级别。
:::

## 5、数据库慢查询应该如何排查和优化
慢查询排查要先定位慢在哪里，再根据执行计划、索引、SQL 写法、数据量和系统负载进行优化。

::: details 详情
### 排查入口

常见入口包括：

- 慢查询日志。
- APM 监控。
- 数据库监控面板。
- 接口耗时日志。
- 用户反馈。

不要只看单条 SQL，还要结合调用频率和业务影响。

### 查看执行计划

可以使用：

```sql
explain select * from users where phone = '13800000000';
```

重点关注：

- 是否走索引。
- 扫描行数。
- join 顺序。
- 是否出现 filesort。
- 是否出现临时表。

### 常见优化方式

- 给高频过滤条件建立合适索引。
- 避免 `select *`。
- 避免在索引字段上使用函数。
- 优化分页方式。
- 拆分复杂 SQL。
- 减少不必要的 join。
- 控制返回数据量。

### 索引失效示例

```sql
select * from users where left(phone, 3) = '138';
```

对索引字段使用函数，可能导致索引无法有效利用。

可以改成范围查询或增加额外字段。

### 注意事项

- 加索引前要看字段选择性和写入成本。
- 慢查询可能不是 SQL 本身，而是锁等待、连接池耗尽或磁盘 IO 问题。
- 优化要用真实数据量验证。
- 修改 SQL 后要重新看执行计划和接口耗时。
:::

## 6、联合索引的最左前缀原则是什么
联合索引是多个字段组成的索引。最左前缀原则是指查询条件需要从联合索引最左侧字段开始连续匹配，才能更好地利用索引。

::: details 详情
### 示例

假设有联合索引：

```sql
create index idx_user_age_city on users(age, city, name);
```

这个索引可以较好支持：

```sql
where age = 18
where age = 18 and city = '成都'
where age = 18 and city = '成都' and name = 'Tom'
```

因为这些条件都从最左字段 `age` 开始。

### 不符合最左前缀

下面的查询可能无法充分利用这个联合索引：

```sql
where city = '成都'
where name = 'Tom'
where city = '成都' and name = 'Tom'
```

因为跳过了最左侧字段 `age`。

### 范围查询影响

如果联合索引中某个字段使用范围查询，后续字段的索引利用可能受影响：

```sql
where age > 18 and city = '成都'
```

具体是否继续利用后续字段，要结合数据库版本和执行计划判断。

### 设计联合索引

设计时要考虑：

- 高频查询条件。
- 字段选择性。
- 等值查询和范围查询顺序。
- 排序和分组需求。

### 面试要点

- 联合索引不是多个单列索引的简单相加。
- 字段顺序非常重要。
- 是否命中索引要看执行计划。
- 不要为了所有查询都建联合索引，写入成本和维护成本也要考虑。
:::
