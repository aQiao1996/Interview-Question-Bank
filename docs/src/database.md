# 数据库

这里用于整理数据库相关面试题，后续按 SQL、索引、事务、锁、MVCC、慢查询优化、分页优化、连接池、Redis 等方向追加。

## 1、数据库索引为什么能提升查询速度
数据库索引是一种辅助数据结构，可以帮助数据库更快定位数据，避免对整张表做全表扫描。常见关系型数据库中，B+ 树索引非常常见。

::: details 详情
### 没有索引时

如果没有索引，查询可能需要逐行扫描：

```sql
select * from users where name = 'Tom';
```

数据量越大，全表扫描成本越高。

### 有索引时

如果 `name` 字段有索引，数据库可以先在索引结构中查找 `Tom` 对应的位置，再回表读取完整数据。

这类似查字典时先看目录，而不是从第一页翻到最后一页。

### B+ 树索引特点

B+ 树适合数据库索引，原因包括：

- 树高较低，减少磁盘 IO。
- 叶子节点有序，适合范围查询。
- 非叶子节点只保存索引键，能容纳更多索引项。
- 查询、插入、删除都有较稳定的复杂度。

### 索引的代价

索引不是越多越好：

- 会占用额外存储空间。
- 插入、更新、删除时需要维护索引。
- 低选择性字段建索引收益可能不高。
- 复杂查询中索引可能不被优化器选择。

### 常见适合建索引的字段

- 经常出现在 `where` 条件中的字段。
- 经常用于 `join` 的字段。
- 经常用于 `order by` 或 `group by` 的字段。
- 高区分度字段，例如手机号、订单号。

### 面试要点

- 索引通过减少扫描范围提升查询性能。
- 索引会降低写入性能，需要权衡。
- 联合索引要注意最左前缀原则。
- 是否使用索引要结合执行计划分析。
:::

## 2、数据库事务的 ACID 分别是什么
事务是一组要么全部成功、要么全部失败的数据库操作。ACID 是事务的四个核心特性：原子性、一致性、隔离性和持久性。

::: details 详情
### Atomicity 原子性

原子性表示事务中的操作要么全部执行成功，要么全部回滚。

例如转账：

```txt
A 扣款 100
B 加款 100
```

这两个操作必须一起成功或一起失败。

### Consistency 一致性

一致性表示事务执行前后，数据库都要满足业务约束和数据约束。

例如账户余额不能无故增加或减少，唯一索引不能被破坏。

### Isolation 隔离性

隔离性表示多个事务并发执行时，彼此之间不能产生不符合隔离级别预期的干扰。

常见并发问题包括：

- 脏读。
- 不可重复读。
- 幻读。

不同隔离级别解决的问题不同。

### Durability 持久性

持久性表示事务提交后，数据变更应该被持久保存，即使数据库宕机，也应该能通过日志等机制恢复。

### 事务示例

```sql
begin;

update accounts set balance = balance - 100 where id = 1;
update accounts set balance = balance + 100 where id = 2;

commit;
```

如果中间失败，需要执行 `rollback`。

### 面试要点

- 原子性关注全部成功或全部失败。
- 一致性关注业务约束和数据约束不被破坏。
- 隔离性关注并发事务之间的影响。
- 持久性关注提交后的数据可靠保存。
:::

## 3、MVCC 是什么，解决了什么问题
MVCC 是多版本并发控制，通过为数据维护多个版本，让读写操作在很多场景下可以并发执行，减少锁竞争，提高数据库并发能力。

::: details 详情
### 为什么需要 MVCC

如果读写都依赖互斥锁，并发性能会很差：

```txt
读阻塞写
写阻塞读
```

MVCC 的核心目标是让读操作尽量不阻塞写操作，写操作也尽量不阻塞普通一致性读。

### 基本思想

数据更新时，不是简单覆盖旧值，而是形成新版本。

事务读取数据时，根据自己的可见性规则读取某个版本。

在 InnoDB 中，MVCC 通常和：

- undo log。
- Read View。
- 事务 ID。
- 隐藏字段。

这些机制有关。

### 快照读和当前读

快照读读取的是事务可见的数据版本，例如普通 `select`。

当前读读取的是最新数据，并可能加锁，例如：

```sql
select * from users where id = 1 for update;
```

### 解决的问题

MVCC 可以减少：

- 读写阻塞。
- 锁竞争。
- 并发事务之间的不必要等待。

在不同隔离级别下，MVCC 的可见性规则会不同。

### 注意事项

- MVCC 不代表完全不需要锁。
- 更新、删除、当前读仍可能加锁。
- 长事务会导致旧版本长期无法清理。
- MVCC 和隔离级别、undo log、事务可见性要结合理解。
:::

## 4、数据库事务隔离级别有哪些
事务隔离级别用于控制并发事务之间的可见性和影响范围。隔离级别越高，并发异常越少，但性能和并发能力可能越低。

::: details 详情
### 常见并发问题

- 脏读：读到其他事务未提交的数据。
- 不可重复读：同一事务内两次读取同一行，结果不同。
- 幻读：同一事务内两次范围查询，结果行数不同。

### Read Uncommitted

读未提交，可能读到其他事务未提交的数据。

它隔离性最低，实际业务中很少使用。

### Read Committed

读已提交，只能读到其他事务已经提交的数据。

可以避免脏读，但可能出现不可重复读。

### Repeatable Read

可重复读，保证同一事务内多次读取同一数据结果一致。

MySQL InnoDB 默认隔离级别通常是 Repeatable Read。

### Serializable

串行化，隔离级别最高，事务像串行执行一样。

它可以避免大多数并发问题，但并发性能最低。

### 对比

```txt
Read Uncommitted -> Read Committed -> Repeatable Read -> Serializable
隔离性逐渐增强，并发能力通常逐渐降低
```

### 面试要点

- 隔离级别是性能和一致性的权衡。
- 不同数据库对隔离级别的实现细节可能不同。
- MySQL InnoDB 的 Repeatable Read 通常结合 MVCC 和锁机制。
- 业务中要根据一致性要求选择隔离级别，而不是盲目使用最高级别。
:::

## 5、数据库慢查询应该如何排查和优化
慢查询排查要先定位慢在哪里，再根据执行计划、索引、SQL 写法、数据量和系统负载进行优化。

::: details 详情
### 排查入口

常见入口包括：

- 慢查询日志。
- APM 监控。
- 数据库监控面板。
- 接口耗时日志。
- 用户反馈。

不要只看单条 SQL，还要结合调用频率和业务影响。

### 查看执行计划

可以使用：

```sql
explain select * from users where phone = '13800000000';
```

重点关注：

- 是否走索引。
- 扫描行数。
- join 顺序。
- 是否出现 filesort。
- 是否出现临时表。

### 常见优化方式

- 给高频过滤条件建立合适索引。
- 避免 `select *`。
- 避免在索引字段上使用函数。
- 优化分页方式。
- 拆分复杂 SQL。
- 减少不必要的 join。
- 控制返回数据量。

### 索引失效示例

```sql
select * from users where left(phone, 3) = '138';
```

对索引字段使用函数，可能导致索引无法有效利用。

可以改成范围查询或增加额外字段。

### 注意事项

- 加索引前要看字段选择性和写入成本。
- 慢查询可能不是 SQL 本身，而是锁等待、连接池耗尽或磁盘 IO 问题。
- 优化要用真实数据量验证。
- 修改 SQL 后要重新看执行计划和接口耗时。
:::

## 6、联合索引的最左前缀原则是什么
联合索引是多个字段组成的索引。最左前缀原则是指查询条件需要从联合索引最左侧字段开始连续匹配，才能更好地利用索引。

::: details 详情
### 示例

假设有联合索引：

```sql
create index idx_user_age_city on users(age, city, name);
```

这个索引可以较好支持：

```sql
where age = 18
where age = 18 and city = '成都'
where age = 18 and city = '成都' and name = 'Tom'
```

因为这些条件都从最左字段 `age` 开始。

### 不符合最左前缀

下面的查询可能无法充分利用这个联合索引：

```sql
where city = '成都'
where name = 'Tom'
where city = '成都' and name = 'Tom'
```

因为跳过了最左侧字段 `age`。

### 范围查询影响

如果联合索引中某个字段使用范围查询，后续字段的索引利用可能受影响：

```sql
where age > 18 and city = '成都'
```

具体是否继续利用后续字段，要结合数据库版本和执行计划判断。

### 设计联合索引

设计时要考虑：

- 高频查询条件。
- 字段选择性。
- 等值查询和范围查询顺序。
- 排序和分组需求。

### 面试要点

- 联合索引不是多个单列索引的简单相加。
- 字段顺序非常重要。
- 是否命中索引要看执行计划。
- 不要为了所有查询都建联合索引，写入成本和维护成本也要考虑。
:::

## 7、Redis 常见数据类型有哪些
Redis 是内存数据库，常用于缓存、计数、分布式锁、排行榜、消息队列等场景。理解常见数据类型有助于选择合适的数据结构。

::: details 详情
### String

String 是最基础的数据类型，适合：

- 缓存字符串或 JSON。
- 计数器。
- 分布式锁。

```txt
SET user:1 '{"name":"Tom"}'
INCR page:view
```

### Hash

Hash 适合存储对象字段：

```txt
HSET user:1 name Tom age 18
HGET user:1 name
```

适合用户信息、配置项等结构化数据。

### List

List 是有序列表，适合：

- 简单队列。
- 最新消息列表。
- 时间线。

```txt
LPUSH queue task1
RPOP queue
```

### Set

Set 是无序不重复集合，适合：

- 去重。
- 标签。
- 共同好友。
- 集合运算。

### ZSet

ZSet 是有序集合，每个元素有分数，适合：

- 排行榜。
- 权重排序。
- 延迟队列。

```txt
ZADD rank 100 user1
ZRANGE rank 0 10 WITHSCORES
```

### 注意事项

- 选择数据类型要结合访问模式。
- 大 key 会影响 Redis 性能。
- 缓存数据要设置合理过期时间。
- Redis 虽快，但不能忽略持久化、内存和淘汰策略。
:::

## 8、数据库中行锁和表锁有什么区别
行锁和表锁都是数据库并发控制中的锁类型。行锁锁定具体数据行，表锁锁定整张表，二者在并发能力和开销上不同。

::: details 详情
### 表锁

表锁会锁住整张表。

优点：

- 实现简单。
- 管理开销小。

缺点：

- 锁粒度大。
- 并发能力差。
- 容易阻塞其他事务。

适合写入较少或需要整表操作的场景。

### 行锁

行锁只锁住满足条件的数据行。

优点：

- 锁粒度小。
- 并发能力更好。

缺点：

- 管理开销更高。
- 可能出现死锁。
- 如果查询没有命中索引，可能扩大锁范围。

### 示例

```sql
update users set name = 'Tom' where id = 1;
```

如果 `id` 是索引，通常只锁定对应行。

如果条件没有索引，数据库可能扫描更多记录，导致锁范围变大。

### 死锁

两个事务互相等待对方释放锁，就可能产生死锁。

常见减少方式：

- 保持相同加锁顺序。
- 缩短事务时间。
- 尽量使用索引定位数据。
- 避免大事务。

### 面试要点

- 表锁粒度大，开销小，并发差。
- 行锁粒度小，开销高，并发好。
- 行锁是否精准依赖索引和执行计划。
- 锁问题要结合事务、隔离级别和 SQL 执行计划分析。
:::

## 9、Redis RDB 和 AOF 有什么区别
RDB 和 AOF 都是 Redis 持久化方式。RDB 是定期生成数据快照，AOF 是记录写命令日志。

::: details 详情
### RDB

RDB 会在某个时间点生成 Redis 数据快照。

优点：

- 文件紧凑。
- 恢复速度较快。
- 适合备份。

缺点：

- 两次快照之间的数据可能丢失。
- 生成快照时可能带来性能开销。

### AOF

AOF 会记录每次写命令：

```txt
SET user:1 Tom
INCR count
```

Redis 重启时重新执行 AOF 中的命令恢复数据。

优点：

- 数据安全性通常更高。
- 可以配置不同刷盘策略。

缺点：

- 文件可能更大。
- 恢复速度可能比 RDB 慢。
- 需要 AOF rewrite 压缩文件。

### 刷盘策略

AOF 常见策略：

- always：每次写入都刷盘，最安全但性能差。
- everysec：每秒刷盘，常用折中方案。
- no：由操作系统决定，性能好但风险更高。

### 如何选择

实际生产中可以同时开启 RDB 和 AOF：

- RDB 适合备份和快速恢复。
- AOF 适合降低数据丢失风险。

### 面试要点

- RDB 是快照。
- AOF 是写命令日志。
- RDB 恢复快但可能丢更多数据。
- AOF 数据更完整但文件更大、恢复可能更慢。
:::

## 10、数据库深分页为什么慢，如何优化
深分页通常指 `limit offset, size` 中 offset 很大。数据库需要扫描并丢弃大量前置数据，再返回目标页，因此会变慢。

::: details 详情
### 深分页示例

```sql
select * from orders order by id limit 1000000, 20;
```

这表示跳过前 1000000 条，再取 20 条。

即使只返回 20 条，数据库也可能需要扫描大量数据。

### 为什么慢

主要原因：

- offset 越大，扫描越多。
- 排序可能产生额外成本。
- 回表次数可能很多。
- 大量数据被读取后又丢弃。

### 基于游标分页

可以改成基于上一次最后 ID 查询：

```sql
select * from orders
where id > 1000000
order by id
limit 20;
```

这种方式可以利用索引范围扫描，适合连续翻页。

### 延迟关联

也可以先查主键，再回表：

```sql
select * from orders
where id in (
  select id from orders order by id limit 1000000, 20
);
```

实际写法要结合数据库优化器，避免子查询反而变慢。

### 注意事项

- 游标分页不适合直接跳到任意页。
- 排序字段要有索引。
- 分页结果要有稳定排序条件。
- 超大页码可以限制访问或使用搜索引擎方案。
:::

## 11、数据库连接池有什么作用
数据库连接池用于复用数据库连接，避免每次请求都创建和销毁连接，从而降低开销并控制数据库连接数量。

::: details 详情
### 为什么需要连接池

创建数据库连接通常有成本：

- TCP 连接。
- 鉴权。
- 初始化会话。
- 分配数据库资源。

如果每个请求都新建连接，会增加延迟，也容易把数据库连接打满。

### 连接池做什么

连接池通常负责：

- 创建一定数量的连接。
- 复用空闲连接。
- 控制最大连接数。
- 处理等待队列。
- 检测连接是否可用。
- 释放长期空闲连接。

### 常见参数

常见配置包括：

- 最小连接数。
- 最大连接数。
- 获取连接超时时间。
- 空闲连接超时时间。
- 连接最大生命周期。

### 连接泄漏

如果业务代码获取连接后没有释放，连接池会逐渐耗尽。

典型表现：

- 接口越来越慢。
- 获取连接超时。
- 数据库连接数打满。

### 注意事项

- 最大连接数不是越大越好，数据库承载能力有限。
- 多实例部署时要计算总连接数。
- 慢查询会长期占用连接。
- 连接池指标要纳入监控，例如活跃连接、空闲连接和等待队列。
:::

## 12、哪些情况会导致索引失效
索引失效是指 SQL 虽然有索引，但优化器没有使用或无法高效使用索引，最终导致全表扫描或扫描范围过大。

::: details 详情
### 常见原因

常见导致索引失效的情况包括：

- 对索引列使用函数或表达式。
- 隐式类型转换。
- 联合索引不满足最左前缀。
- `like` 以通配符开头。
- 使用 `or` 时部分条件没有索引。
- 查询条件选择性太低。
- 统计信息不准确。

### 函数和表达式

例如：

```sql
select * from user where year(created_at) = 2026;
```

如果 `created_at` 上有索引，这种写法可能无法有效使用索引。

可以改成范围查询：

```sql
select * from user
where created_at >= '2026-01-01'
  and created_at < '2027-01-01';
```

### 隐式类型转换

如果字段是字符串，但查询时传数字：

```sql
select * from user where phone = 13800138000;
```

数据库可能发生隐式转换，影响索引使用。

### 如何排查

可以通过 `EXPLAIN` 查看：

- 是否使用索引。
- 使用了哪个索引。
- 扫描行数。
- 过滤比例。
- 是否出现临时表或文件排序。

### 注意事项

- 索引失效不一定是语法错误，更多是优化器选择问题。
- 数据量小的时候不走索引也可能是合理选择。
- 建索引前要结合查询频率、选择性和写入成本评估。
:::

## 13、什么是覆盖索引
覆盖索引是指查询需要的字段都能从索引中直接获得，不需要再回到数据行读取完整记录。它可以减少随机 IO，提高查询性能。

::: details 详情
### 基本概念

假设有联合索引：

```sql
create index idx_user_name_age on user(name, age);
```

查询：

```sql
select name, age from user where name = 'Tom';
```

如果查询字段都在索引中，数据库可以只扫描索引完成查询。

### 回表是什么

在 InnoDB 中，普通二级索引通常保存索引列和主键值。

如果查询字段不在索引里，就需要先通过二级索引找到主键，再通过主键回到聚簇索引读取完整行，这就是回表。

覆盖索引可以减少回表。

### 使用场景

覆盖索引适合：

- 高频查询。
- 返回字段较少。
- 查询条件和返回字段比较稳定。
- 分页列表等读多场景。

例如列表页只展示 `id`、`title`、`created_at`，可以设计联合索引覆盖查询。

### 注意事项

- 覆盖索引不是字段越多越好，索引过宽会增加存储和写入成本。
- 查询中 `select *` 通常不利于覆盖索引。
- 联合索引字段顺序要结合过滤、排序和返回字段设计。
- 是否覆盖要通过执行计划确认。
:::

## 14、Redis 缓存穿透、击穿和雪崩有什么区别
缓存穿透、击穿和雪崩都是缓存失效或未命中导致数据库压力增大的问题，但触发原因和治理方式不同。

::: details 详情
### 缓存穿透

缓存穿透是指查询一个缓存和数据库都不存在的数据。

如果大量请求查询不存在的 key，请求会一直打到数据库。

常见解决方式：

- 缓存空值。
- 使用布隆过滤器。
- 参数校验，拦截明显非法请求。
- 对异常访问限流。

### 缓存击穿

缓存击穿是指某个热点 key 过期瞬间，大量请求同时打到数据库。

常见解决方式：

- 热点 key 不设置过短过期时间。
- 互斥锁重建缓存。
- 后台异步刷新缓存。
- 使用逻辑过期时间。

### 缓存雪崩

缓存雪崩是指大量 key 同时失效，或者缓存服务整体不可用，导致请求集中打到数据库。

常见解决方式：

- 过期时间加随机值。
- 多级缓存。
- Redis 高可用。
- 限流和降级。
- 提前预热核心缓存。

### 注意事项

- 缓存空值要设置较短过期时间，避免脏数据长期存在。
- 互斥锁要设置超时，避免死锁。
- 热点缓存重建失败时要有降级数据。
- 缓存问题本质上要结合流量、热点和数据库承载能力一起设计。
:::

## 15、如何看懂 SQL 执行计划
SQL 执行计划用于展示数据库准备如何执行一条查询，包括访问表的顺序、使用的索引、扫描行数和额外操作。它是排查慢查询的重要工具。

::: details 详情
### 基本用法

可以使用：

```sql
explain select * from user where name = 'Tom';
```

不同数据库输出字段不同，但核心都是帮助判断查询是否高效。

### 重点字段

以 MySQL 为例，常关注：

- `type`：访问类型，例如 `const`、`ref`、`range`、`ALL`。
- `possible_keys`：可能使用的索引。
- `key`：实际使用的索引。
- `rows`：预计扫描行数。
- `filtered`：过滤比例。
- `Extra`：额外信息，例如 `Using index`、`Using filesort`。

### 常见信号

需要重点关注：

- `type` 为 `ALL`，可能是全表扫描。
- `rows` 很大，扫描范围过大。
- 没有使用预期索引。
- 出现 `Using filesort`。
- 出现 `Using temporary`。

这些不一定都是问题，但需要结合数据量和业务频率判断。

### 优化思路

看到执行计划后，可以考虑：

- 调整查询条件。
- 新增或调整联合索引。
- 避免 `select *`。
- 减少函数和隐式转换。
- 拆分复杂 SQL。
- 更新统计信息。

### 注意事项

- 执行计划是优化器估算结果，不一定完全等于真实执行情况。
- 同一 SQL 在不同数据量下可能选择不同索引。
- 慢查询要结合执行时间、锁等待和返回行数一起分析。
- 不要只为了让执行计划好看而盲目加索引。
:::

## 16、为什么数据库索引常用 B+ 树
B+ 树适合数据库索引，因为它层高低、磁盘 IO 次数少、范围查询友好，并且所有数据通常存储在叶子节点，叶子节点之间有顺序链表。

::: details 详情
### B+ 树特点

B+ 树常见特点：

- 多路平衡搜索树。
- 非叶子节点主要存索引键。
- 数据记录通常在叶子节点。
- 叶子节点按顺序连接。
- 树高较低。

这些特点适合磁盘或页式存储。

### 为什么不是二叉树

二叉树每个节点最多两个子节点。

数据量大时树高会比较高，查询路径更长，磁盘 IO 次数也可能更多。

B+ 树一个节点可以存多个 key，扇出更大，树高更低。

### 范围查询

B+ 树叶子节点有序连接。

范围查询时，找到起点后可以沿叶子节点链表顺序扫描。

例如：

```sql
where age between 20 and 30
```

这类查询很适合 B+ 树索引。

### 注意事项

- 索引字段越大，单页能放的 key 越少，树可能更高。
- 随机写入可能导致页分裂。
- 主键过长会影响二级索引体积。
- B+ 树适合范围查询，但不是所有查询都能高效命中索引。
:::

## 17、SQL 注入是什么，如何防御
SQL 注入是攻击者把恶意 SQL 片段拼接进查询语句中，改变原本 SQL 语义，从而绕过登录、读取数据、修改数据甚至破坏数据库。

::: details 详情
### 典型问题

危险写法：

```js
const sql = `select * from user where name = '${name}'`;
```

如果用户输入：

```txt
' or '1'='1
```

就可能改变查询逻辑。

### 防御方式

最重要的是使用参数化查询：

```js
db.query("select * from user where name = ?", [name]);
```

数据库会把参数当作数据处理，而不是 SQL 语句的一部分。

### 其他措施

还可以结合：

- ORM 参数绑定。
- 输入格式校验。
- 最小数据库权限。
- 禁止错误信息直接暴露 SQL。
- 对高风险操作加审计。
- WAF 作为辅助防护。

### 常见误区

只做字符串替换或简单转义并不可靠。

黑名单很容易被绕过，例如编码、注释、大小写变化等。

### 注意事项

- 排序字段、表名这类不能参数化的位置，要使用白名单映射。
- 防 SQL 注入不能只靠前端校验。
- 数据库账号不要使用过高权限。
- 日志中要避免记录敏感 SQL 参数明文。
:::

## 18、分布式事务常见方案有哪些
分布式事务用于保证跨多个服务或多个数据库操作的一致性。常见方案包括 2PC、TCC、本地消息表、事务消息和 Saga。

::: details 详情
### 2PC

2PC 是两阶段提交。

第一阶段询问各参与者是否可以提交，第二阶段统一提交或回滚。

优点是一致性较强，缺点是协调者依赖重、阻塞风险高、性能较差。

### TCC

TCC 包括：

- Try：预留资源。
- Confirm：确认执行。
- Cancel：取消释放资源。

它需要业务实现三套接口，适合对一致性要求高、业务可控的场景。

### 本地消息表

业务数据和消息记录写在同一个本地事务中。

之后异步投递消息，下游消费后完成最终一致。

它适合很多互联网业务，但需要处理重试、幂等和消息堆积。

### Saga

Saga 把长事务拆成多个本地事务，每一步都有对应补偿动作。

适合流程较长、允许最终一致的业务。

### 注意事项

- 分布式事务优先从业务上避免，而不是默认引入复杂方案。
- 任何异步最终一致方案都要考虑幂等、重试和补偿。
- 高价值资金类场景需要更严格的对账和审计。
- 方案选择要结合一致性要求、性能、开发成本和故障处理能力。
:::

## 19、Redis 过期删除策略有哪些
Redis 过期删除策略用于清理设置了过期时间的 key。它通常结合惰性删除、定期删除和内存淘汰策略一起工作。

::: details 详情
### 惰性删除

惰性删除是指访问 key 时检查是否过期。

如果过期，就删除并返回不存在。

优点是简单，不会主动消耗太多 CPU。

缺点是如果过期 key 一直不被访问，可能继续占用内存。

### 定期删除

Redis 会周期性抽样检查一部分设置了过期时间的 key。

如果发现过期，就删除。

这种方式可以主动清理过期数据，但不会一次性扫描所有 key，避免阻塞主线程。

### 内存淘汰

如果内存达到上限，Redis 会根据淘汰策略删除部分 key。

常见策略包括：

- `noeviction`
- `allkeys-lru`
- `volatile-lru`
- `allkeys-lfu`
- `volatile-ttl`
- `allkeys-random`

### 业务影响

过期策略会影响缓存行为：

- 大量 key 同时过期可能造成缓存雪崩。
- 热点 key 过期可能造成缓存击穿。
- 内存淘汰可能让未过期 key 被删除。

### 注意事项

- 缓存过期时间建议加随机抖动。
- 重要缓存要有重建和降级方案。
- 设置 maxmemory 和淘汰策略前要理解业务后果。
- 需要监控内存使用、过期 key 数量和淘汰 key 数量。
:::

## 20、数据库主从复制是如何工作的
主从复制是指主库负责写入，从库复制主库数据并承担部分读请求。它常用于提升读能力、容灾备份和读写分离。

::: details 详情
### 基本流程

以 MySQL 为例，主从复制大致流程：

1. 主库提交事务并写入 binlog。
2. 从库 IO 线程拉取主库 binlog。
3. 从库写入 relay log。
4. 从库 SQL 线程重放 relay log。
5. 从库数据逐渐追上主库。

### 复制模式

常见复制模式：

- 异步复制。
- 半同步复制。
- 组复制。

异步复制性能较好，但主库宕机时可能丢失尚未同步到从库的数据。

半同步复制会等待至少一个从库确认接收日志，可靠性更高，但延迟也可能增加。

### 常见问题

主从复制可能出现：

- 主从延迟。
- 从库中断。
- 数据不一致。
- 主从切换后数据回放问题。
- 读到旧数据。

### 注意事项

- 强一致读应走主库或使用一致性策略。
- 要监控复制延迟和复制线程状态。
- 主从切换要有明确流程和演练。
- 读写分离时要处理写后读旧数据的问题。
:::

## 21、Redis Sentinel 有什么作用
Redis Sentinel 用于监控 Redis 主从节点、发现主节点故障并自动完成故障转移，从而提升 Redis 主从架构的可用性。

::: details 详情
### 核心能力

Sentinel 主要提供：

- 监控主从节点是否可用。
- 故障检测。
- 自动选举新的主节点。
- 通知客户端新的主节点地址。
- 维护主从拓扑。

它本身通常需要部署多个实例，避免单点。

### 故障转移

当 Sentinel 判断主节点不可用后，会：

1. 多个 Sentinel 达成故障判断。
2. 选举一个 Sentinel 作为领导者。
3. 从从节点中选择一个提升为新主节点。
4. 让其他从节点复制新主节点。
5. 通知客户端主节点变化。

### 主观下线和客观下线

主观下线是单个 Sentinel 认为节点不可用。

客观下线是多个 Sentinel 达成一致，认为主节点确实不可用。

这样可以降低误判风险。

### 注意事项

- Sentinel 负责高可用，不负责数据分片。
- 主从异步复制下，故障切换仍可能丢少量数据。
- 客户端要支持 Sentinel 协议，能感知主节点变化。
- Sentinel、Redis 节点和网络分区都要纳入监控。
:::

## 22、数据库慢查询日志有什么作用
慢查询日志用于记录执行时间超过阈值的 SQL，帮助定位性能瓶颈、缺失索引、扫描行数过多和锁等待等问题。

::: details 详情
### 记录什么

慢查询日志通常会记录：

- SQL 语句。
- 执行时间。
- 扫描行数。
- 返回行数。
- 锁等待时间。
- 执行时间点。
- 用户和数据库。

不同数据库字段略有不同。

### 如何使用

排查慢查询时可以：

1. 找出耗时高、频率高的 SQL。
2. 使用执行计划分析索引使用情况。
3. 查看扫描行数和返回行数差异。
4. 判断是否存在锁等待。
5. 优化索引、SQL 或表结构。

优化应优先处理高频且影响核心链路的 SQL。

### 常见原因

- 缺少合适索引。
- 索引失效。
- 返回数据过多。
- 深分页。
- 大表 join。
- 锁等待。
- 数据量增长后原 SQL 不再适用。

### 注意事项

- 慢查询阈值要结合业务设置，不是越低越好。
- 慢查询日志可能包含敏感参数，要注意脱敏和权限。
- 只看单次耗时不够，还要看频率和总耗时。
- SQL 优化后要通过压测或线上指标验证效果。
:::

## 23、数据库备份和恢复如何设计
数据库备份和恢复用于在误删、故障、数据损坏或灾难发生时找回数据。备份方案不仅要能备份，更要能验证恢复。

::: details 详情
### 备份类型

常见备份包括：

- 全量备份。
- 增量备份。
- 日志备份。
- 快照备份。
- 异地备份。

全量备份简单但成本高，增量和日志备份可以降低存储和时间成本。

### 恢复目标

设计备份时要明确：

- RPO：最多能接受丢失多少数据。
- RTO：最多能接受多久恢复服务。

不同业务对 RPO 和 RTO 要求不同，核心交易系统通常要求更严格。

### 恢复演练

备份文件存在不代表一定能恢复。

需要定期演练：

- 从备份恢复到测试环境。
- 校验数据完整性。
- 验证恢复耗时。
- 验证应用是否能连接恢复后的数据库。
- 记录演练过程和问题。

### 注意事项

- 备份要加密存储，并控制访问权限。
- 备份要放在和主库不同的故障域。
- 误删数据时要避免新写入覆盖恢复窗口。
- 恢复流程要文档化，不能只依赖少数人经验。
:::

## 24、数据库事务隔离级别有哪些
事务隔离级别用于控制多个事务并发执行时彼此可见的数据范围。隔离级别越高，一致性越强，但并发性能通常越低。

::: details 详情
### 常见问题

事务并发可能带来：

- 脏读：读到其他事务未提交的数据。
- 不可重复读：同一事务内两次读取同一行结果不同。
- 幻读：同一事务内两次范围查询返回的行数不同。

隔离级别就是为了解决这些问题。

### 四种隔离级别

常见隔离级别包括：

- Read Uncommitted：可能发生脏读。
- Read Committed：避免脏读，但可能不可重复读。
- Repeatable Read：避免脏读和不可重复读。
- Serializable：最高隔离级别，事务串行执行效果。

不同数据库的具体实现可能不同。

### MySQL 默认级别

InnoDB 默认通常是 `Repeatable Read`。

它通过 MVCC 和锁机制解决很多并发一致性问题。

在部分场景下，还需要间隙锁和临键锁处理范围查询。

### 注意事项

- 隔离级别不是越高越好，要结合业务一致性要求。
- 核心资金类业务通常需要更严格的并发控制。
- 普通查询业务可以接受较低隔离级别以提高并发。
- 面试时要说明现象、隔离级别和数据库实现差异。
:::

## 25、MVCC 是什么
MVCC 是多版本并发控制，用于让读写操作尽量不互相阻塞。它通过保存数据的多个版本，让事务在一致性视图下读取数据。

::: details 详情
### 核心思想

当一行数据被修改时，数据库不会立刻覆盖旧值，而是保留版本信息。

读取事务根据自己的快照版本，决定看到哪一版数据。

这样普通读操作通常不会阻塞写操作。

### 作用

MVCC 主要解决：

- 读写并发冲突。
- 快照读一致性。
- 提高并发性能。

它是很多数据库实现一致性读的重要基础。

### 和锁的关系

MVCC 主要用于“快照读”。

当前读或写操作仍然可能需要加锁，例如：

- `SELECT ... FOR UPDATE`
- `UPDATE`
- `DELETE`

所以 MVCC 不是完全不要锁。

### 注意事项

- 不同数据库的 MVCC 实现细节不同。
- 版本太多会带来额外存储和清理成本。
- 长事务会导致旧版本难以清理。
- 面试时要结合 undo log、事务快照和隔离级别一起说。
:::

## 26、数据库连接池有什么作用
数据库连接池用于复用数据库连接，避免每次请求都创建和销毁连接。它可以降低连接建立成本，并控制应用对数据库的并发压力。

::: details 详情
### 为什么需要连接池

创建数据库连接通常包含：

- TCP 连接。
- 认证。
- 初始化会话状态。
- 分配数据库资源。

如果每个请求都新建连接，性能和数据库资源都会受到影响。

### 关键参数

常见参数包括：

- 最大连接数。
- 最小空闲连接数。
- 连接获取超时时间。
- 空闲连接回收时间。
- 连接最大生命周期。
- 健康检查 SQL。

这些参数要结合数据库承载能力和应用实例数一起设置。

### 常见问题

连接池配置不当可能导致：

- 连接数过高压垮数据库。
- 连接数过低导致请求排队。
- 连接泄漏。
- 慢 SQL 占满连接池。
- 数据库重启后连接不可用。

排查时要同时看应用连接池指标和数据库连接数。

### 注意事项

- 连接池不是越大越好。
- 应用实例扩容后，总连接数也会增加。
- 使用完连接要及时归还。
- 长事务和慢查询会占用连接，影响其他请求。
:::

## 27、分库分表什么时候需要考虑
分库分表用于解决单库单表在数据量、写入压力或存储成本上达到瓶颈的问题。它能提升扩展性，但也会显著增加系统复杂度。

::: details 详情
### 什么时候考虑

通常在以下情况考虑分库分表：

- 单表数据量非常大。
- 索引变慢、查询性能下降。
- 写入压力超过单库承载能力。
- 数据增长速度超过垂直扩容能力。
- 需要按业务线或租户隔离数据。

### 常见方案

常见拆分方式包括：

- 按范围分片。
- 按哈希分片。
- 按时间分片。
- 按业务维度分片。

不同方式在扩容、热点和查询模式上各有取舍。

### 代价

分库分表会带来：

- 跨分片查询困难。
- 分布式事务复杂。
- 全局排序和分页困难。
- 主键生成复杂。
- 运维和迁移成本上升。

所以它不是默认选择，先看单库优化是否足够。

### 注意事项

- 需要提前设计分片键。
- 迁移要考虑双写、回填和校验。
- 热点数据可能让分片失衡。
- 面试时要说清楚为什么拆、怎么拆、怎么查、怎么迁。
:::

## 28、乐观锁适合什么场景
乐观锁适合读多写少、冲突概率较低的场景。它不会在读取时阻塞其他事务，而是在更新时检查数据是否被别人修改过。

::: details 详情
### 基本思路

常见实现是版本号字段：

```sql
UPDATE product
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 10;
```

如果影响行数为 0，说明数据已经被其他事务修改，需要重试或返回失败。

### 适合场景

乐观锁适合：

- 用户资料更新。
- 配置编辑。
- 库存扣减中的低冲突场景。
- 表单提交防覆盖。
- 状态机流转。

它能减少锁等待，提高并发能力。

### 和悲观锁对比

乐观锁假设冲突少，提交时再检查。

悲观锁假设冲突多，读取或修改时先加锁。

冲突频繁时，乐观锁会导致大量重试，反而影响性能。

### 注意事项

- 更新条件必须包含版本号或时间戳。
- 更新失败要有明确处理策略。
- 自动重试要限制次数。
- 复杂业务要确认重试是否会产生副作用。
:::

## 29、悲观锁适合什么场景
悲观锁适合写冲突概率高、必须严格串行处理的场景。它会在操作前锁定数据，避免其他事务同时修改。

::: details 详情
### 基本用法

常见方式是使用当前读加锁：

```sql
SELECT *
FROM account
WHERE id = 1
FOR UPDATE;
```

在事务提交或回滚前，其他事务通常不能修改这行数据。

### 适合场景

悲观锁适合：

- 账户余额扣减。
- 库存高冲突扣减。
- 抢占资源。
- 状态必须严格流转。
- 不能接受重试覆盖的核心业务。

它更强调一致性，但并发性能可能下降。

### 风险

悲观锁可能带来：

- 锁等待。
- 死锁。
- 事务时间过长。
- 吞吐下降。
- 影响范围过大。

因此事务内逻辑要尽量短。

### 注意事项

- 加锁查询必须命中索引，避免锁范围扩大。
- 事务中不要做远程调用或耗时操作。
- 要设置合理的锁等待超时时间。
- 出现死锁时要能重试或回滚。
:::
