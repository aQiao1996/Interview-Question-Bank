# 安全

这里用于整理安全相关面试题，后续按 XSS、CSRF、CSP、点击劫持、JWT、OAuth、SSO、权限模型、接口签名、防重放、供应链安全等方向追加。

## 1、什么是 XSS，如何防御
XSS 是跨站脚本攻击，攻击者把恶意脚本注入到页面中，当用户访问页面时脚本被浏览器执行，从而窃取信息或执行恶意操作。

::: details 详情
### 常见类型

- 存储型 XSS：恶意内容存入数据库，用户访问时触发。
- 反射型 XSS：恶意内容来自 URL 或请求参数，服务端直接拼回页面。
- DOM 型 XSS：前端脚本把不可信数据写入 DOM 导致执行。

### 危害

XSS 可能导致：

- 窃取 token 或 cookie。
- 冒充用户操作。
- 修改页面内容。
- 劫持跳转。
- 读取页面敏感信息。

### 防御方式

核心原则是不要把不可信内容当作代码执行。

常见手段包括：

- 输出时做 HTML 转义。
- 避免直接使用 `innerHTML`。
- 对富文本使用白名单过滤。
- 设置 Cookie 的 `HttpOnly`。
- 使用 CSP 限制脚本来源。
- 对 URL、样式、事件属性等特殊上下文单独处理。

### 示例

危险写法：

```js
container.innerHTML = userInput;
```

更安全的写法：

```js
container.textContent = userInput;
```

### 注意事项

- 输入校验不能替代输出编码。
- 富文本不能简单转义，要做标签和属性白名单。
- `HttpOnly` 不能阻止 XSS 发生，但能降低 cookie 被读取的风险。
- CSP 是纵深防御，不应作为唯一防线。
:::

## 2、什么是 CSRF，如何防御
CSRF 是跨站请求伪造攻击。攻击者诱导已登录用户访问恶意页面，恶意页面利用用户浏览器自动携带 Cookie 的特性，向目标站点发起非用户本意的请求。

::: details 详情
### 攻击条件

CSRF 通常需要满足：

- 用户已经登录目标网站。
- 登录态依赖 Cookie 自动携带。
- 目标接口缺少额外校验。
- 攻击者能诱导用户打开恶意页面。

### 示例

恶意页面中可能放置：

```html
<img src="https://bank.example.com/transfer?to=attacker&amount=1000" />
```

如果目标站点只依赖 Cookie 判断用户身份，就可能执行非用户本意的操作。

### 防御方式

常见防御包括：

- CSRF Token。
- SameSite Cookie。
- 校验 Origin 或 Referer。
- 关键操作二次确认。
- 避免 GET 接口执行状态变更。

### CSRF Token

服务端生成随机 token，页面请求时带上 token：

```txt
Cookie: session=xxx
Header: X-CSRF-Token: random-token
```

攻击站点通常无法读取目标站点页面中的 token，因此伪造请求无法通过校验。

### SameSite

```http
Set-Cookie: session=xxx; SameSite=Lax
```

`SameSite` 可以限制跨站请求是否携带 Cookie。

### 注意事项

- CSRF 防御重点在服务端，不能只靠前端隐藏按钮。
- XSS 可以绕过很多 CSRF 防御，因此 XSS 和 CSRF 要一起防。
- Token 要和用户会话绑定，并有合理生命周期。
- 敏感操作应避免使用 GET 请求。
:::

## 3、CSP 是什么，能解决什么安全问题
CSP 是内容安全策略，用于限制页面可以加载和执行哪些资源，常用于降低 XSS、数据注入和资源劫持带来的风险。

::: details 详情
### 基本作用

CSP 通过 HTTP 响应头声明安全策略：

```http
Content-Security-Policy: default-src 'self'; script-src 'self'
```

这表示默认只允许加载同源资源，脚本也只允许同源脚本。

### 常见指令

- `default-src`：默认资源加载策略。
- `script-src`：脚本来源。
- `style-src`：样式来源。
- `img-src`：图片来源。
- `connect-src`：接口、WebSocket 等连接来源。
- `frame-ancestors`：限制页面被哪些来源 iframe 嵌入。

### 防御 XSS

如果页面配置了严格的 `script-src`，即使攻击者注入了内联脚本，也可能因为 CSP 限制而无法执行。

例如：

```http
Content-Security-Policy: script-src 'self'
```

这会阻止很多外部恶意脚本和内联脚本执行。

### Report-Only 模式

上线前可以使用：

```http
Content-Security-Policy-Report-Only: default-src 'self'
```

它只上报违规，不真正拦截，适合灰度观察。

### 注意事项

- CSP 是纵深防御，不能替代输入校验和输出编码。
- 策略太宽松效果有限，例如大量使用 `unsafe-inline`。
- 策略太严格可能导致正常资源加载失败。
- 需要结合真实资源域名、监控和上报逐步收紧。
:::

## 4、JWT 由哪几部分组成，使用时要注意什么
JWT 是一种常见的令牌格式，通常用于身份认证和授权信息传递。它由 Header、Payload 和 Signature 三部分组成。

::: details 详情
### 组成结构

JWT 结构如下：

```txt
header.payload.signature
```

### Header

Header 描述令牌类型和签名算法，例如：

```json
{
  "typ": "JWT",
  "alg": "HS256"
}
```

### Payload

Payload 存放声明信息，例如：

```json
{
  "sub": "user-1",
  "role": "admin",
  "exp": 1710000000
}
```

注意 Payload 只是 Base64URL 编码，不是加密，不能存放密码、手机号等敏感信息。

### Signature

Signature 用于校验令牌是否被篡改。

服务端会用密钥或私钥对 Header 和 Payload 签名，验证时重新计算签名并比对。

### 优点

- 服务端可以不存储 session。
- 适合分布式系统。
- 可以携带少量用户声明。
- 跨服务验证方便。

### 注意事项

- JWT 不能随便存敏感数据。
- 要设置合理过期时间。
- 密钥泄露会导致严重风险。
- 服务端必须校验签名和过期时间。
- 退出登录和强制失效需要额外设计黑名单或版本号机制。
:::

## 5、OAuth2 授权码模式是什么
OAuth2 授权码模式是一种常见授权流程，适合 Web 应用和第三方登录场景。它通过授权码中转，避免前端直接暴露长期敏感凭证。

::: details 详情
### 参与角色

OAuth2 中常见角色：

- Resource Owner：资源拥有者，通常是用户。
- Client：第三方应用。
- Authorization Server：授权服务器。
- Resource Server：资源服务器。

### 基本流程

```txt
用户点击登录
-> 跳转授权服务器
-> 用户登录并授权
-> 授权服务器回跳并带上 code
-> Client 后端用 code 换 access_token
-> 使用 access_token 访问资源
```

### 为什么使用授权码

授权码是短期凭证，真正的 token 由后端通过安全通道换取。

这样可以降低 token 直接暴露在浏览器 URL 或前端代码中的风险。

### state 的作用

请求授权时通常会带上 `state`：

```txt
state=random-value
```

回调时服务端校验 `state`，用于防止 CSRF 和登录流程被篡改。

### PKCE

移动端、SPA 等公共客户端通常会配合 PKCE，降低授权码被截获后的风险。

PKCE 会使用：

- `code_verifier`。
- `code_challenge`。

换 token 时必须证明自己持有正确的 `code_verifier`。

### 注意事项

- OAuth2 主要解决授权，不等同于认证。
- 如果需要确认用户身份，通常使用 OIDC。
- 回调地址必须严格白名单校验。
- access token 要设置合理过期时间，并保护好 refresh token。
:::

## 6、RBAC 权限模型是什么
RBAC 是基于角色的访问控制模型，通过“用户-角色-权限”的关系管理系统权限，常用于后台管理系统和企业应用。

::: details 详情
### 核心关系

RBAC 的基本关系是：

```txt
用户 -> 角色 -> 权限
```

例如：

```txt
张三 -> 财务管理员 -> 查看账单、导出账单、审批付款
```

用户不直接绑定大量权限，而是通过角色间接获得权限。

### 常见实体

常见表设计包括：

- 用户表。
- 角色表。
- 权限表。
- 用户角色关联表。
- 角色权限关联表。

### 权限粒度

权限可以分为：

- 菜单权限。
- 页面权限。
- 按钮权限。
- 接口权限。
- 数据权限。

前端常控制菜单和按钮展示，后端必须控制接口和数据访问。

### 数据权限

数据权限用于控制用户能看到哪些数据。

例如：

- 只能看本人数据。
- 只能看本部门数据。
- 可以看全公司数据。

数据权限通常比菜单按钮权限更复杂。

### 注意事项

- 前端权限控制只能改善体验，不能替代后端鉴权。
- 权限变更后要考虑 token、缓存和会话刷新。
- 角色过多会导致权限管理复杂。
- 高风险操作要记录审计日志。
:::

## 7、接口签名有什么作用，如何设计
接口签名用于确认请求没有被篡改，并验证调用方身份，常用于开放 API、支付回调、服务间调用等场景。

::: details 详情
### 为什么需要接口签名

如果请求参数在传输中被篡改，服务端需要能发现。

接口签名通常解决：

- 参数篡改。
- 调用方身份确认。
- 重放攻击风险降低。
- 开放平台接口安全。

### 常见签名参数

请求中通常包含：

```txt
appId
timestamp
nonce
signature
```

服务端根据相同规则重新计算签名并比对。

### 签名内容

签名通常覆盖：

```txt
method + path + query + bodyHash + timestamp + nonce
```

这样攻击者修改请求方法、路径、参数或请求体，签名都会不匹配。

### 签名算法

常见方式：

- HMAC-SHA256。
- RSA 私钥签名、公钥验签。

内部服务或开放平台常用 HMAC，第三方高安全场景可能使用非对称签名。

### 防重放

签名通常要配合：

- 时间戳有效期。
- nonce 去重。
- 请求 ID。

否则攻击者可能重复发送完全相同的合法请求。

### 注意事项

- 签名密钥不能出现在前端代码中。
- 参数排序规则要明确。
- body 参与签名时要统一序列化规则。
- 签名失败要记录日志，但不要泄露密钥或完整敏感参数。
:::

## 8、前端项目如何做供应链安全治理
供应链安全关注项目依赖、构建工具、发布流程和第三方脚本是否可信。前端项目依赖多、构建链路长，因此供应链风险需要持续治理。

::: details 详情
### 常见风险

- 依赖包被投毒。
- 安装恶意相似包。
- lockfile 被篡改。
- 构建脚本执行恶意代码。
- CDN 第三方脚本被污染。
- 私有 registry 权限泄露。

### 依赖治理

常见措施：

- 提交 lockfile。
- 固定包管理器版本。
- 使用私有镜像源或可信 registry。
- 定期运行依赖安全扫描。
- 谨慎升级高风险依赖。
- 删除长期不用的依赖。

### 安装脚本风险

很多 npm 包可以在安装时执行脚本。

可以在 CI 或高安全环境中考虑：

```bash
pnpm install --ignore-scripts
```

但要注意部分依赖确实需要安装脚本，需要评估兼容性。

### 第三方脚本

外部脚本要注意：

- 是否必须引入。
- 是否固定版本。
- 是否配置 SRI。
- 是否受 CSP 限制。

### 注意事项

- 不要随意复制执行陌生命令。
- CI 中的 token 权限要最小化。
- 发布 npm 包要开启双因素认证。
- 供应链安全不是一次扫描，而是持续流程。
:::

## 9、常见 Web 安全响应头有哪些
安全响应头用于让浏览器启用额外安全策略，降低 XSS、点击劫持、信息泄露和跨域滥用风险。它们不能替代业务安全校验，但能作为重要防线。

::: details 详情
### 常见响应头

- `Content-Security-Policy`：限制脚本、样式、图片等资源来源。
- `X-Frame-Options`：限制页面是否允许被 iframe 嵌入。
- `Strict-Transport-Security`：强制浏览器使用 HTTPS。
- `Referrer-Policy`：控制请求携带的来源信息。
- `Permissions-Policy`：限制摄像头、定位、麦克风等浏览器能力。
- `X-Content-Type-Options: nosniff`：禁止浏览器 MIME 嗅探。

### CSP

CSP 是防御 XSS 的重要手段之一。

例如：

```http
Content-Security-Policy: default-src 'self'; script-src 'self'
```

它可以限制脚本只能从当前站点加载。

### 点击劫持防护

可以使用：

```http
X-Frame-Options: DENY
```

或在 CSP 中使用：

```http
Content-Security-Policy: frame-ancestors 'none'
```

### 注意事项

- 安全响应头要结合业务逐步灰度，避免误伤第三方资源。
- CSP 可以先使用 Report-Only 模式观察违规情况。
- HTTPS 站点建议配置 HSTS，但要确认全站 HTTPS 已经稳定。
- 响应头是浏览器侧防护，接口权限和数据校验仍必须后端完成。
:::

## 10、富文本场景如何防御 XSS
富文本允许用户输入带格式的内容，比普通文本更容易出现 XSS 风险。防御重点是白名单过滤、输出安全渲染和限制脚本执行能力。

::: details 详情
### 风险来源

富文本中可能包含：

- `<script>` 标签。
- 事件属性，例如 `onclick`。
- `javascript:` URL。
- 恶意 iframe。
- SVG 中的脚本能力。
- 样式中的危险表达式或外链。

如果直接使用 `innerHTML` 渲染用户内容，就可能执行恶意脚本。

### 白名单过滤

富文本不能只做简单字符串替换，应该使用成熟的 HTML sanitizer。

过滤策略通常是白名单：

- 允许安全标签，例如 `p`、`strong`、`ul`、`li`、`a`。
- 允许安全属性，例如 `href`、`title`。
- 禁止事件属性。
- 禁止危险协议。
- 限制 iframe、style、svg 等高风险能力。

### 渲染时注意

前端渲染富文本时要：

- 只渲染经过清洗的 HTML。
- 链接加上 `rel="noopener noreferrer"`。
- 图片地址做协议和域名限制。
- 配合 CSP 限制脚本来源。

### 注意事项

- 前端过滤不能替代服务端过滤，存储前和展示前都要考虑安全。
- 黑名单很容易漏掉绕过方式，优先使用白名单。
- 管理后台预览富文本也要防 XSS。
- 不同业务允许的标签不同，不要直接开放完整 HTML 能力。
:::

## 11、前端登录态存储要注意什么安全问题
前端登录态通常涉及 Cookie、Token、localStorage 和 sessionStorage。核心问题是如何在用户体验、跨端能力、XSS 风险和 CSRF 风险之间做权衡。

::: details 详情
### 常见存储方式

常见方案包括：

- Cookie：可由浏览器自动携带，适合传统会话。
- HttpOnly Cookie：前端 JS 无法读取，能降低 XSS 窃取风险。
- localStorage：使用方便，但可被 XSS 读取。
- sessionStorage：页面会话级存储，关闭标签页后清理。
- 内存变量：刷新丢失，但暴露面更小。

### Cookie 安全属性

使用 Cookie 存登录态时，建议配置：

```http
Set-Cookie: sid=xxx; HttpOnly; Secure; SameSite=Lax
```

含义：

- `HttpOnly`：禁止 JS 读取。
- `Secure`：只在 HTTPS 下发送。
- `SameSite`：降低 CSRF 风险。

### Token 存储风险

Token 放在 localStorage 中，一旦页面存在 XSS，攻击者就可能读取 Token。

如果业务必须使用 Token，可以配合：

- 缩短有效期。
- 使用 Refresh Token 轮换。
- 绑定设备或环境风险。
- 重要操作二次验证。

### 注意事项

- 不要把敏感登录态写入 URL，URL 容易进入日志和 Referer。
- 退出登录要清理本地状态和服务端会话。
- XSS 和 CSRF 要一起考虑，不能只防其中一个。
- 前端存储只影响暴露风险，最终鉴权必须由后端完成。
:::

## 12、文件上传功能有哪些安全风险
文件上传是高风险入口，攻击者可能上传恶意脚本、超大文件、伪造类型文件或包含敏感内容的文件。安全设计要覆盖校验、存储、访问和处理全链路。

::: details 详情
### 常见风险

- 上传可执行脚本。
- 伪造 MIME Type 或文件后缀。
- 上传超大文件耗尽存储或带宽。
- 图片、PDF 中隐藏恶意内容。
- 文件名包含路径穿越字符。
- 上传后文件被公开访问导致信息泄露。

### 前端能做什么

前端可以做体验层校验：

- 限制文件类型。
- 限制文件大小。
- 展示上传进度。
- 上传前预览。
- 提示不支持的格式。

但前端校验可以被绕过，不能作为安全边界。

### 后端必须做什么

后端需要重新校验：

- 校验真实文件类型。
- 限制大小和数量。
- 重命名文件，避免使用原始文件名作为存储路径。
- 文件存储和应用服务隔离。
- 私有文件使用鉴权下载。
- 高风险文件做杀毒或内容扫描。

### 注意事项

- 上传目录不要允许执行脚本。
- 公开访问文件要使用独立域名，降低 Cookie 和主站风险。
- 图片处理库也可能有漏洞，要保持更新。
- 文件下载时设置正确的 `Content-Type` 和 `Content-Disposition`。
:::

## 13、什么是 SSRF，如何防御
SSRF 是服务端请求伪造攻击，攻击者诱导服务端向内网地址、云元数据地址或敏感服务发起请求，从而读取或操作本不该暴露的资源。

::: details 详情
### 常见场景

SSRF 常出现在服务端根据用户输入访问 URL 的功能中：

- 图片抓取。
- 网页截图。
- URL 预览。
- 文件下载。
- Webhook 回调。
- 第三方资源导入。

如果服务端没有限制目标地址，攻击者可能让它访问内网服务。

### 风险

SSRF 可能导致：

- 访问内网管理系统。
- 读取云厂商元数据。
- 扫描内网端口。
- 绕过公网访问控制。
- 攻击 Redis、Elasticsearch 等未授权服务。

### 防御方式

常见防御：

- 使用域名或 IP 白名单。
- 禁止访问内网 IP、回环地址和链路本地地址。
- 禁止跳转到非白名单地址。
- 限制协议，只允许 HTTP/HTTPS。
- 设置请求超时和响应大小限制。
- 服务端网络隔离，限制出站访问。

### 注意事项

- 只校验字符串不够，要解析 DNS 后校验最终 IP。
- 要防 DNS Rebinding。
- 跟随重定向后也要重新校验目标地址。
- 云环境要保护元数据服务访问权限。
:::

## 14、开放重定向漏洞是什么
开放重定向是指应用把用户可控的 URL 作为跳转目标，且没有做有效校验，攻击者可以构造可信域名链接把用户跳转到钓鱼网站或恶意页面。

::: details 详情
### 常见形式

例如登录后跳转：

```txt
https://example.com/login?redirect=https://evil.com
```

如果服务端或前端直接读取 `redirect` 并跳转，就可能形成开放重定向。

### 风险

开放重定向常用于：

- 钓鱼攻击。
- 绕过部分安全检查。
- 诱导用户输入账号密码。
- OAuth 回调链路攻击。
- 利用可信域名提升链接可信度。

虽然它看起来只是跳转问题，但在登录、支付、授权场景中风险更高。

### 防御方式

常见防御：

- 只允许相对路径跳转。
- 使用跳转白名单。
- 禁止协议相对 URL，例如 `//evil.com`。
- 对 URL 做标准化解析后再校验。
- OAuth 场景严格匹配 redirect_uri。

推荐使用内部路径：

```txt
/dashboard
```

而不是完整外部 URL。

### 注意事项

- 不要只用 `startsWith` 简单判断域名，容易被绕过。
- 要处理 URL 编码、大小写、反斜杠等变形。
- 外部跳转页面可以增加二次确认提示。
- 登录态相关跳转要特别谨慎。
:::

## 15、验证码功能如何设计才安全
验证码用于区分正常用户和自动化请求，常见于登录、注册、短信发送、找回密码和高风险操作。安全设计要同时考虑有效性、可用性和抗绕过能力。

::: details 详情
### 常见类型

常见验证码包括：

- 图片验证码。
- 短信验证码。
- 邮箱验证码。
- 滑块或行为验证码。
- 二次验证。

不同验证码适合不同风险等级，不能一套方案覆盖所有场景。

### 关键设计

验证码需要关注：

- 有效期。
- 使用次数。
- 绑定业务场景。
- 绑定手机号、邮箱或会话。
- 失败次数限制。
- 发送频率限制。

短信验证码尤其要防止被刷，避免造成资损。

### 常见绕过

攻击者可能尝试：

- 重放旧验证码。
- 枚举验证码。
- 绕过前端校验直接调接口。
- 批量请求短信接口。
- 使用打码平台。

因此验证码校验必须在服务端完成。

### 注意事项

- 验证码通过后应立即失效或标记已使用。
- 不要在响应中返回验证码明文。
- 高风险操作应结合风控、限流和设备指纹。
- 验证码只是防护手段之一，不能替代账号安全策略。
:::

## 16、越权漏洞有哪些类型，如何防御
越权漏洞是指用户访问或操作了自己无权访问的数据或功能。常见类型包括水平越权、垂直越权和未授权访问。

::: details 详情
### 水平越权

水平越权是同一权限等级的用户访问了别人的资源。

例如用户 A 修改请求中的 `userId`，读取到用户 B 的订单。

防御重点是服务端按当前登录用户校验资源归属。

### 垂直越权

垂直越权是低权限用户访问了高权限功能。

例如普通用户调用管理员删除接口。

防御重点是服务端做角色、权限码或策略校验。

### 未授权访问

未授权访问是没有登录或没有凭证也能访问接口。

常见原因：

- 接口漏加鉴权中间件。
- 内部接口暴露到公网。
- 默认账号密码未修改。
- 对象存储权限配置错误。

### 防御方式

- 所有敏感接口服务端统一鉴权。
- 查询和修改资源时校验数据归属。
- 权限校验不要只依赖前端菜单和按钮。
- 使用统一权限中间件或策略引擎。
- 对高风险接口做审计日志。

### 注意事项

- 前端隐藏按钮不能防越权。
- ID 改成 UUID 只能降低猜测概率，不能替代权限校验。
- 批量接口也要逐条校验权限。
- 权限模型变化后要回归测试核心接口。
:::

## 17、密码重置功能如何设计才安全
密码重置是账号体系中的高风险功能。设计时要防止重置链接泄露、验证码被爆破、账号枚举和重置后旧会话继续可用等问题。

::: details 详情
### 基本流程

常见流程：

1. 用户提交账号、邮箱或手机号。
2. 服务端生成一次性重置凭证。
3. 通过邮箱或短信发送给用户。
4. 用户在有效期内完成验证。
5. 设置新密码。
6. 使旧凭证和旧会话失效。

整个流程要避免暴露账号是否存在。

### 重置凭证

重置凭证应满足：

- 随机性足够强。
- 有较短有效期。
- 只能使用一次。
- 服务端存储 hash，而不是明文。
- 绑定用户和业务场景。

不要把用户 ID、时间戳等可预测信息直接拼成 token。

### 防爆破

需要限制：

- 单账号发送频率。
- 单 IP 请求频率。
- 验证失败次数。
- 同一设备或同一手机号的请求次数。

高风险场景可以加入验证码或二次验证。

### 注意事项

- 响应文案不要提示账号是否存在。
- 密码重置成功后应通知用户。
- 重置后可以让旧登录态失效。
- 日志中不要记录重置 token 或验证码明文。
:::

## 18、多因素认证 MFA 如何设计
MFA 是在密码之外增加额外验证因素，例如短信验证码、邮箱验证码、TOTP、硬件安全密钥或生物识别，用来降低密码泄露后的账号风险。

::: details 详情
### 常见因素

认证因素通常分为：

- 知识因素：密码、PIN。
- 持有因素：手机、邮箱、安全密钥。
- 生物因素：指纹、人脸。

MFA 的价值在于攻击者即使拿到密码，也还需要第二个因素。

### 常见方式

常见 MFA 方式：

- 短信验证码。
- 邮箱验证码。
- TOTP 动态码。
- Push 确认。
- WebAuthn/FIDO2 安全密钥。

短信实现简单，但容易受 SIM 卡劫持、短信拦截和运营商风险影响。

### 触发策略

MFA 不一定每次都强制触发。

可以在高风险场景触发：

- 新设备登录。
- 异地登录。
- 修改密码。
- 修改绑定手机号。
- 提现、支付等敏感操作。
- 管理后台高权限操作。

### 注意事项

- 要设计备用恢复方式，但恢复流程也必须安全。
- 绑定和解绑 MFA 都应二次验证。
- 高权限账号应强制开启 MFA。
- MFA 失败次数要限制，防止爆破。
:::

## 19、API 网关可以做哪些安全能力
API 网关位于客户端和后端服务之间，可以集中处理鉴权、限流、黑白名单、协议转换、日志审计和风险拦截，是系统安全边界的一部分。

::: details 详情
### 常见能力

API 网关常做：

- 统一鉴权。
- Token 校验。
- IP 黑白名单。
- 请求限流。
- 参数大小限制。
- CORS 控制。
- WAF 或基础攻击拦截。
- 访问日志和审计。

这些能力集中在网关可以减少各业务服务重复实现。

### 鉴权和转发

网关可以校验用户 Token 或应用签名。

校验通过后，网关可以把用户 ID、租户 ID、角色等可信上下文转发给后端服务。

后端服务仍应做必要的业务权限校验，不能完全依赖网关。

### 限流

网关限流可以按：

- IP。
- 用户。
- 应用。
- 租户。
- 接口。
- 地域。

不同接口应有不同限流策略。

### 注意事项

- 网关是关键入口，要高可用。
- 网关鉴权失败要返回统一错误，不要泄露内部信息。
- 内网服务也要防止绕过网关直接访问。
- 敏感操作仍要在业务服务中做权限和数据归属校验。
:::

## 20、日志脱敏应该如何设计
日志脱敏用于防止密码、Token、手机号、身份证号、银行卡号等敏感信息进入日志系统，降低泄露风险和合规风险。

::: details 详情
### 哪些数据需要脱敏

常见敏感信息包括：

- 密码。
- Token 和 Cookie。
- 手机号。
- 邮箱。
- 身份证号。
- 银行卡号。
- 地址。
- 生物识别信息。
- 业务密钥和签名。

不同业务和地区法规对敏感信息定义可能不同。

### 脱敏方式

常见方式：

- 掩码：`138****8000`。
- 哈希：用于排查关联但不暴露明文。
- 删除字段。
- 白名单记录字段。
- 对结构化日志统一过滤。

相比黑名单，日志字段白名单通常更可控。

### 处理位置

脱敏可以发生在：

- 业务代码打印前。
- 日志框架中间件。
- 网关或代理层。
- 日志采集 Agent。
- 日志平台入库前。

越靠近源头越安全。

### 注意事项

- 不要把请求头中的 Authorization、Cookie 原样输出。
- 异常堆栈和第三方错误也可能包含敏感数据。
- 脱敏规则要有测试，防止字段变更后漏脱敏。
- 高权限日志查询也要有审计和访问控制。
:::
